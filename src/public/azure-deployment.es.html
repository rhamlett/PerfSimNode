🚀 Azure Guía de despliegue

Despliega Performance Problem Simulator para Azure App Service usando GitHub Acciones con autenticación OIDC

Resumen

Esta guía te guía para desplegar el Performance Problem Simulator para Azure App Service usando GitHub Acciones con autenticación OpenID Connect (OIDC). OIDC elimina la necesidad de almacenar credenciales como secretos.

Requisitos previos

  • Azure suscripción con permisos para crear recursos
  • Azure permisos AD/Entra ID para crear registros de aplicaciones
  • GitHub cuenta
  • Azure CLIs instalados (opcional, para el método CLI)

Lo que vas a crear

RecursoPropósito
Azure App ServiceAloja la aplicación Performance Problem Simulator
Azure Registro en la aplicación ADIdentidad para GitHub Acciones OIDC
Credencial FederadaEnlaces GitHub repositorio a Azure AD
Asignación de rolesConcede permisos de despliegue

1 Crea Azure App Service

Opción A: Azure Portal

  1. Navegar a los Servicios de Aplicaciones
    • Ir a Azure Portal
    • Busca "App Services" y selecciónalo
    • Clic + Create
  2. Configurar lo básico
    AmbientaciónValor
    SuscripciónTu suscripción
    Grupo de RecursosCrear nuevos o usar ya existentes
    Nombreyour-app-name (debe ser única a nivel global)
    PublicarCódigo
    Pila de tiempo de ejecuciónSelecciona el tiempo de ejecución de tu aplicación
    Sistema operativoWindows o Linux (según corresponda a tu entorno de ejecución)
    RegiónTu región preferida
  3. Configurar App Service plan
    AmbientaciónRecomendación
    SKUB1 básico o superior

    ⚠️ Importante: La categoría Free (F1) no soporta WebSockets, que son necesarios para actualizaciones en tiempo real de SignalR paneles.

  4. Revisar y crear - Haz clic para crear el recurso

Opción B: Azure CLI

# Login to Azure
az login

# Set variables
RESOURCE_GROUP="your-resource-group"
APP_NAME="your-app-name"
LOCATION="eastus"
APP_SERVICE_PLAN="your-app-plan"
RUNTIME="your-runtime"  # e.g., "dotnet:8", "node:20-lts", "python:3.11"

# Create resource group
az group create --name $RESOURCE_GROUP --location $LOCATION

# Create App Service Plan (B1 for WebSocket support)
az appservice plan create \
  --name $APP_SERVICE_PLAN \
  --resource-group $RESOURCE_GROUP \
  --location $LOCATION \
  --sku B1

# Create Web App with your runtime
az webapp create \
  --name $APP_NAME \
  --resource-group $RESOURCE_GROUP \
  --plan $APP_SERVICE_PLAN \
  --runtime $RUNTIME

Habilitar WebSockets

Tras la creación, activa WebSockets para SignalR comunicación en tiempo real:

  1. Ve a tu App Service → Configuración → Configuraciones generales
  2. Set Sockets web a On
  3. Clic Salvar

O a través de CLI:

az webapp config set \
  --name $APP_NAME \
  --resource-group $RESOURCE_GROUP \
  --web-sockets-enabled true

2 Crear Azure Registro de la App de AD

GitHub Acciones utiliza OpenID Connect (OIDC) para autenticarse con Azure sin almacenar credenciales.

Crear el registro de la app

  1. Navegar a Registros de Aplicaciones
    • Ir a Azure Portal
    • Busca "Registros de aplicaciones" y selecciona
    • Clic + Nueva matrícula
  2. Configurar registro
    AmbientaciónValor
    Nombregithub-your-app-deploy (nombre descriptivo para tu identidad de despliegue)
    Tipos de cuentas soportadosCuentas en este directorio organizativo solo
    URI de redirecciónDeja vacío
  3. Haz clic en Registrarse

Anota las identificaciones

De la Resumen página, copia estos valores (los necesitarás para GitHub secretos):

ValorGitHub Nombre secreto
ID de aplicación (cliente)AZURE_CLIENT_ID
ID de directorio (inquilino)AZURE_TENANT_ID

Obtén ID de suscripción

  1. Ir a Suscripciones en el Azure Portal
  2. Selecciona tu suscripción
  3. Copiar el ID de suscripción → Esto es AZURE_SUBSCRIPTION_ID

3 Configurar credenciales federadas

Las credenciales federadas permiten que GitHub Acciones se autentiquen sin almacenar secretos.

  1. Accede a tu Registro de la Aplicación
    • Ir a Certificados y secretos
    • Seleccionar Credenciales federadas pestaña
    • Clic + Añadir credencial
  2. Configurar la credencial
    AmbientaciónValor
    Escenario de credenciales federadasGitHub Acciones que despliegan Azure recursos
    OrganizaciónTu GitHub nombre de usuario u organización
    RepositorioNombre de tu repositorio
    Tipo de entidadRama
    GitHub nombre de la ramamain
    Nombregithub-main-branch
  3. Haz clic en Añadir

💡 Nota: Si también quieres desplegar desde pull requests u otras ramas, añade credenciales federadas adicionales con el tipo de entidad correspondiente.

4Conceder permisos Azure

El Registro de la App necesita permiso para desplegarse en tu App Service.

Asignar rol de colaborador

  1. Navega hacia tu App Service
    • Ir a Control de acceso (IAM)
    • Clic + Añadir → Añadir asignación de roles
  2. Configurar asignación de roles
    AmbientaciónValor
    FunciónColaborador
    Asignar acceso aUsuario, grupo o principal de servicio
    MiembrosBusca tu nombre de registro de la app
  3. Haz clic en Revisar + asignar

5 Configurar GitHub Secretos

Clonar el repositorio

  1. Clonar el repositorio
    • Ve al repositorio Performance Problem Simulator para tu pila
    • Clic Código y clonar para crear tu propia copia

Añadir GitHub Secretos

  1. Navega a Configuración del repositorio
    • Ve a tu repositorio en GitHub
    • Clic Escenarios → Secretos y variables → Acciones
    • Clic Nuevo secreto de repositorio
  2. Añadir los siguientes secretos
    Nombre secretoValorDescripción
    AZURE_CLIENT_IDxxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxxID de cliente de registro de aplicaciones
    AZURE_TENANT_IDxxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxxAzure Directorio/ID de inquilino de AD
    AZURE_SUBSCRIPTION_IDxxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxxAzure ID de suscripción

💡 No se necesita secreto del cliente:OIDC credenciales federadas eliminan la necesidad de almacenar un secreto del cliente. Las tres identificaciones anteriores son suficientes.

Actualizar el flujo de trabajo (si es necesario)

Actualización .github/workflows/deploy.yml Con tu nombre App Service y versión en tiempo de ejecución:

env:
  AZURE_WEBAPP_NAME: your-app-service-name  # Your App Service name
  # Add any runtime-specific version variables as needed

6 Despliegue

Despliegue automático

El despliegue se activa automáticamente cuando presionas en el main Sucursal:

git add .
git commit -m "Your changes"
git push origin main

Despliegue manual

  1. Ve a tu repositorio en GitHub
  2. Clic Acciones pestaña
  3. Seleccionar Desplegarse a Azure App Service
  4. Clic Ejecutar flujo de trabajo → Ejecutar flujo de trabajo

Verificar despliegue

  1. Abrir la URL de la aplicación: https://<your-app-name>.azurewebsites.net
  2. Panel de control de Verify: Las métricas en tiempo real deberían actualizarse mediante SignalR, el estado debería mostrar "Conectado"
  3. Endpoint de salud de la prueba: 
    curl https://<your-app-name>.azurewebsites.net/api/health

Configuración opcional

Puedes personalizar el comportamiento de la aplicación usando Azure App Service variables de entorno.

Tasa de Prueba de Salud

Controla con qué frecuencia el servidor envía sondas de salud para medir la latencia de las solicitudes. Todas las sondas se enrutan a través del frontend Azure para capturar latencias realistas de extremo a extremo.

Variable de entorno

Nombre de la variable Descripción Default
HEALTH_PROBE_RATE Intervalo de sonda en milisegundos. Mínimo 100ms. 200

Configuración mediante Azure CLI

# Slow down probes if CLR profiler shows overlapping requests
az webapp config appsettings set --name $APP_NAME --resource-group $RESOURCE_GROUP \
  --settings HEALTH_PROBE_RATE=400

Tiempo de espera en reposo

Cuando la aplicación está inactiva (sin conexiones al panel ni solicitudes de pruebas de carga), las sondas de salud se suspenden automáticamente para reducir el tráfico de red innecesario al frontend de Azurey a la telemetría Application Insights .

Variable de entorno

Nombre de la variable Descripción Default
IDLE_TIMEOUT_MINUTES Minutos de inactividad antes de suspender las sondas de salud. La actividad se reanuda automáticamente cuando se abre el panel de control o se recibe cualquier solicitud. 20

Configuración mediante Azure CLI

# Extend idle timeout to 30 minutes
az webapp config appsettings set --name $APP_NAME --resource-group $RESOURCE_GROUP \
  --settings IDLE_TIMEOUT_MINUTES=30

Pie de página personalizado

Establece un mensaje personalizado en el pie de página que aparezca en todas las páginas. El pie de página soporta HTML, lo que permite incluir enlaces.

Variable de entorno

Nombre de la variable Descripción
PAGE_FOOTER Contenido HTML para la segunda línea del pie de página. Si no está activado, solo se muestra la descripción de la app y la información de la build.

Valor de ejemplo

Created by <a href="https://yoursite.com">Your Team</a> for training purposes

Configuración mediante Azure CLI

# Set the footer via Azure CLI
az webapp config appsettings set --name $APP_NAME --resource-group $RESOURCE_GROUP \
  --settings 'PAGE_FOOTER=Created by <a href="https://yoursite.com">Your Team</a> for training purposes'

Ambientación a través de Azure Portal

  1. Navega hasta tu App Service en el Portal Azure
  2. Ir a Escenarios → Variables de entorno
  3. Clic + Añadir
  4. Nombre del conjunto: PAGE_FOOTER
  5. Valor fijado: Contenido del pie de página HTML
  6. Clic Solicita, entonces Confirmar

GitHub Enlace al Repositorio

Opcionalmente, muestra un enlace GitHub Repositorio en el menú de navegación de la barra lateral. El enlace está oculto por defecto. Cuando se proporcionan ambas variables de entorno, el enlace aparecerá en el menú y apuntará a https://github.com/{GITHUB_USER_NAME}/{GITHUB_REPO_NAME}.

Variables ambientales

Nombre de la variable Descripción
GITHUB_USER_NAME GitHub nombre de usuario o de la organización. Es necesario mostrar el enlace.
GITHUB_REPO_NAME GitHub nombre del repositorio. Es necesario mostrar el enlace.

Configuración mediante Azure CLI

# Set the GitHub link via Azure CLI
az webapp config appsettings set --name $APP_NAME --resource-group $RESOURCE_GROUP \
  --settings GITHUB_USER_NAME=your-username GITHUB_REPO_NAME=your-repo-name

Ambientación a través de Azure Portal

  1. Navega hasta tu App Service en el Portal Azure
  2. Ir a Escenarios → Variables de entorno
  3. Clic + Añadir y nombre del conjunto: GITHUB_USER_NAME, Valor: tu nombre de usuario u organización GitHub
  4. Clic + Añadir y nombre del conjunto: GITHUB_REPO_NAME, Valor: nombre de tu repositorio
  5. Clic Solicita, entonces Confirmar

Resolución de problemas

SignalR falla la conexión

Síntomas: El panel muestra "Desconectado", los gráficos no se actualizan en tiempo real

Soluciones:

  • Asegúrese de que los WebSockets estén habilitados en App Service Configuración
  • Actualizar de nivel Gratuito a Básico o superior
  • Comprueba si la afinidad ARR está activada (Ajustes → Configuración → Configuración General)

OIDC Falla la autenticación

Síntomas:GitHub Acciones fallan en el paso "Iniciar sesión en Azure"

Comprobar:

  • Verifica que los tres secretos estén correctamente configurados
  • Confirma que el tema de la credencial federada coincide con tu repositorio/rama: repo:USERNAME/REPO:ref:refs/heads/main
  • Asegúrate de que el registro de aplicaciones tenga un papel de Colaborador en la App Service

La aplicación se bloquea al arrancar

Síntomas: La app muestra "Error de solicitud" o no responde

Comprobar:

  • Ver flujo de registro en Azure Portal (App Service → Flujo de registro)
  • Activar el registro de aplicaciones en App Service → App Service Registros
  • Verifica que la versión en tiempo de ejecución coincida con los requisitos de tu aplicación
  • Comprobar si faltan variables de configuración o de entorno

# Stream live logs
az webapp log tail --name $APP_NAME --resource-group $RESOURCE_GROUP

# View deployment logs
az webapp log deployment show --name $APP_NAME --resource-group $RESOURCE_GROUP

Referencia rápida - Línea de CLI

Script de configuración completo (variables de personalización en la parte superior):

# Full setup (customize variables at top)
RESOURCE_GROUP="your-resource-group"
APP_NAME="your-app-name"
LOCATION="eastus"
RUNTIME="your-runtime"  # e.g., "dotnet:8", "node:20-lts", "python:3.11"
GITHUB_REPO="YOUR_USERNAME/YOUR_REPO"

# Create resources
az group create -n $RESOURCE_GROUP -l $LOCATION
az appservice plan create -n "${APP_NAME}-plan" -g $RESOURCE_GROUP -l $LOCATION --sku B1
az webapp create -n $APP_NAME -g $RESOURCE_GROUP -p "${APP_NAME}-plan" --runtime $RUNTIME
az webapp config set -n $APP_NAME -g $RESOURCE_GROUP --web-sockets-enabled true

# Create App Registration and get IDs
APP_ID=$(az ad app create --display-name "github-${APP_NAME}-deploy" --query appId -o tsv)
az ad sp create --id $APP_ID
TENANT_ID=$(az account show --query tenantId -o tsv)
SUB_ID=$(az account show --query id -o tsv)

# Create federated credential
az ad app federated-credential create --id $APP_ID --parameters "{
  \"name\": \"github-main-branch\",
  \"issuer\": \"https://token.actions.githubusercontent.com\",
  \"subject\": \"repo:${GITHUB_REPO}:ref:refs/heads/main\",
  \"audiences\": [\"api://AzureADTokenExchange\"]
}"

# Assign permissions
APP_SERVICE_ID=$(az webapp show -n $APP_NAME -g $RESOURCE_GROUP --query id -o tsv)
az role assignment create --assignee $APP_ID --role Contributor --scope $APP_SERVICE_ID

# Output secrets for GitHub
echo "AZURE_CLIENT_ID=$APP_ID"
echo "AZURE_TENANT_ID=$TENANT_ID"
echo "AZURE_SUBSCRIPTION_ID=$SUB_ID"
