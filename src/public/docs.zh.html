CPU 应力模拟

通过不同的操作系统进程产生高 CPU 使用率，通过以下方式 child_process.fork().

工作原理

与主线程中 CPU 的天真烧录（会阻挡Node.js）不同，这种模拟会生成独立的子进程，每个进程运行 crypto.pbkdf2Sync() 陷入紧密循环。操作系统调度器将这些进程分布在 CPU 核心之间。

💡 Node.js见解：CPU密集型工作必须与主事件循环隔离开来。子进程保证真正的多核利用率，且不会阻碍请求处理。

用途

POST /api/simulations/cpu
Content-Type: application/json

{
  "intensity": "high",
  "durationSeconds": 30
}

参数

参数价值观描述
强度“温和” |“高”CPU 应力强度水平
持续时间秒1-300运行模拟多久

预期效果

  • CPU 图块/图表的目标百分比增加
  • 事件循环延迟保持低（工作在子进程中）
  • 由于争用，请求延迟可能会略有增加CPU
  • 多重 node 可见的过程 top/htop
📊 关于 CPU 指标： 仪表盘以250毫秒间隔采样 CPU 原始数据 os.cpus() 数据。这种高分辨率方法即使在空闲时也可能因 GC、卷度探头和容器调度而出现显著波动。专注于 趋势 是在模拟过程中，而不是单个尖峰。生产工具（Azure Monitor）每1分钟采样一次并进行平均值。

诊断工具

  • top / htop - 查看所有节点进程
  • ps aux | grep node - 参见子进程树
  • Azure App Service 诊断→ CPU 深入分析
  • Application Insights → 性能指标

内存压力仿真

分配并保留充满随机数据的内存缓冲区，以模拟内存泄漏。

工作原理

用途 Buffer.alloc() 用于分配被明确释放前被保留的内存。缓冲区填充随机数据以防止V8优化。多个分配可以共存（堆叠行为）。

💡 Node.js见解： V8的垃圾回收器使用代际回收。大分配可能导致更长的 GC 暂停，表现为事件循环延迟峰值。

用途

# Allocate memory
POST /api/simulations/memory
{"sizeMb": 100}

# Release memory (use the returned ID)
DELETE /api/simulations/memory/{id}

参数

参数分布范围描述
sizeMb1-500分配内存（以兆字节为单位）

预期效果

  • 堆内存随分配大小增加
  • RSS内存按比例增加
  • 分配过程中短暂的事件循环延迟峰值（GC）
  • 锯齿形图案， GC 在分配之间运行

诊断工具

  • node --inspect - 通过Chrome DevTools进行堆快照
  • Azure App Service → 内存指标
  • Application Insights → customMetrics for heap stats
  • process.memoryUsage() - 程序访问

事件环阻断模拟

演示同步操作如何阻断Node.js事件循环，使服务器完全无响应。

⚠️ 警告： 在模拟过程中，服务器将完全无响应所有请求。 WebSocket 连接可能会超时。仪表盘会冻结。

工作原理

同步执行 crypto.pbkdf2Sync() 直接在主帖里。所有待处理的I/O、计时器和收到的请求都会排队，直到阻塞完成。

💡 与 CPU 压力的主要区别：CPU stress 使用子进程（服务器保持响应）。事件循环阻断在主线程运行（服务器变得无响应）。查看事件循环延迟指标来区分它们！

用途

POST /api/simulations/eventloop
{"durationSeconds": 5}

需要注意的症状

  • 事件循环延迟峰值以等于阻塞持续时间
  • 所有请求在解封后一起排队并完成
  • WebSocket 心跳停止（仪表盘冻结）
  • 探针点在阻断过程中变红
  • 健康检查失败/超时

诊断工具

  • perf_hooks.monitorEventLoopDelay() - 内置于该应用的指标中
  • Clinic.js 医生 - 分析事件循环问题
  • 0x - 同步热点的火焰图分析

慢请求仿真

利用多种阻断模式模拟慢 HTTP 响应，以观察不同的资源争用行为。

阻挡模式

模式描述受影响的资源
setTimeout非阻断延迟（默认）无——服务器保持响应
libuv饱和 libuv 线程池FS、DNS、加密操作排队
worker生成阻挡工作线程CPU，类似于 .NET ThreadPool
💡 .NET 平行： 该 libuv 以及 worker图案模拟线池饥饿，类似于.NET中耗尽ThreadPool线程的阻断模式sync-over-async。

用途

# Non-blocking (default)
GET /api/simulations/slow?delaySeconds=10

# libuv thread pool saturation
GET /api/simulations/slow?delaySeconds=10&blockingPattern=libuv

# Worker thread blocking
GET /api/simulations/slow?delaySeconds=10&blockingPattern=worker

按模式预期效果

setTimeout（非阻塞）

  • 慢端点需要配置好的延迟时间
  • 探针延迟保持正常
  • 事件循环延迟保持接近零

libuv 线程池饱和度

  • 文件系统操作变慢
  • DNS 查询变得缓慢
  • HTTP 请求仍然正常处理

工作线程阻塞

  • CPU 使用率增加
  • 线数增加
  • 主事件循环不受影响

仪表盘控制

  • 定型模式： setTimeout、libuv 或 worker
  • 延迟： 每个请求需要多长时间（1-300秒）
  • 中场休息： 自动请求之间的时间
  • 最大请求： 要发多少才会停止

使用场景

  • 测试超时配置
  • 模拟线程池饥饿
  • 理解阻塞行为与非阻塞行为

未完成的请求

产生 HTTP 5xx服务器错误，用于测试错误监控、警报和 Azure AppLens 诊断。

工作原理

失败请求模拟会在内部 HTTP 请求，这些请求必然失败，包含500个状态码。每个请求在从17种错误类型中随机抛出异常之前，先执行实际工作（CPU 压力、内存分配、延迟）。

错误类型

错误类型描述
InvalidOperationError（无效操作错误）尝试操作无效
类型错误类型不匹配或空引用
TimeoutError（超时错误）操作时间超过了
内存出错内存分配失败
数据库连接错误无法连接到数据库
认证错误认证失败
网络错误网络通信故障
ServiceUnavailableError（服务不可用错误）后端服务不可用

此外还有9种额外的错误类型，以应对多样化的错误特征。

仪表盘控制

  • 请求数量： 未生成请求次数（1-50）

撞车模拟

故意崩溃测试崩溃恢复和观察重启行为的 Node.js 进程。

⚠️ 警告： 这些操作将终止该过程。 Azure App Service时，进程会自动重启。仪表盘会断开连接再重新连接。

撞击类型

类型终点方法影响
FailFast/crash/failfastprocess.abort()立即SIGABRT，核心导出
堆栈溢出/crash/stackoverflow无限递归RangeError，呼叫堆栈超过。
例外/crash/exceptionthrow Error未处理异常终止
OOM/crash/memory无限分配堆积疲劳， OOM 杀

预期行为

  • 进程立即终止
  • 仪表盘显示断线
  • Azure App Service 自动重启流程
  • 重启后：新的进程ID，清除模拟，新的指标
  • 事件日志显示：“🔄应用程序重启！进程ID更改了......”

诊断工具

可选配置

你可以用 Azure App Service 环境变量自定义应用行为。

健康探针率

控制服务器发送健康探针的频率以测量请求延迟。所有探针都经过 Azure 前端，以捕捉真实的端到端延迟。

环境变量

变量名称 描述 默认
HEALTH_PROBE_RATE 探测间隔以毫秒计。至少100毫秒。 200

通过 Azure CLI设置

# Slow down probes if CLR profiler shows overlapping requests
az webapp config appsettings set --name $APP_NAME --resource-group $RESOURCE_GROUP \
  --settings HEALTH_PROBE_RATE=400

闲置暂停

当应用程序处于空闲状态（无仪表盘连接或负载测试请求）时，健康探针会自动暂停，以减少不必要的网络流量 Azure前端和 Application Insights 遥测数据。

环境变量

变量名称 描述 默认
IDLE_TIMEOUT_MINUTES 暂停健康探针前的几分钟不活动。当仪表盘打开或收到任何请求时，活动会自动恢复。 20

通过 Azure CLI设置

# Extend idle timeout to 30 minutes
az webapp config appsettings set --name $APP_NAME --resource-group $RESOURCE_GROUP \
  --settings IDLE_TIMEOUT_MINUTES=30

自定义页面页脚

设置一个自定义页脚消息，显示在所有页面上。页脚支持 HTML，允许你包含链接。

环境变量

变量名称 描述
PAGE_FOOTER 页脚第二行的HTML内容。如果没有设置，只显示应用描述和构建信息。

示例价值

Created by <a href="https://yoursite.com">Your Team</a> for training purposes

通过 Azure CLI设置

az webapp config appsettings set --name $APP_NAME --resource-group $RESOURCE_GROUP \

语言选择

仪表盘界面可以用以下支持的任何语言显示 Azure 认知服务翻译器.翻译在启动时使用翻译文本 API完成一次。

工作原理

  1. 赛场 UI_LANGUAGE 到支持的语言代码（例如， es, ja, fr)
  2. 启动时，应用会将英文 UI 字符串和 HTML 文档页面翻译成目标语言
  3. 翻译后的文件会缓存在磁盘上——后续使用相同语言的启动会跳过翻译
  4. 如果源英文文件发生变化，缓存将被自动失效并重新翻译

环境变量

变量名称 描述 默认
UI_LANGUAGE 目标语言代码（例如， en, es, ja, fr, de, zh-Hans) en
TRANSLATOR_API_KEY Azure 翻译资源 API 键（非英语必备） （空的）
TRANSLATOR_ENDPOINT Azure 翻译器端点 URL https://api.cognitive.microsofttranslator.com
TRANSLATOR_REGION Azure 翻译器资源的区域 eastus

Azure App Service 上的设置

  1. 在Azure门户创建Azure翻译资源
  2. API从资源的密钥与端点 刀锋
  3. 导航到你的 App Service → 背景设定 → 环境变量
  4. 添加 UI_LANGUAGE, TRANSLATOR_API_KEY， 和 TRANSLATOR_REGION
  5. 咔哒 申请，则 确认 —— 应用将在下一次重启时翻译

注释

  • 当 UI_LANGUAGE=en （默认情况下），不发生翻译，也不需要翻译资源
  • 翻译就是“启动时翻译一次”——没有运行时的翻译开销
  • 技术术语（Azure、 CPU、 API、 JSON等）通过禁止翻译列表受到翻译保护
  • 服务器端事件日志消息包含翻译密钥，因此仪表盘会以所选语言显示它们

建筑

该应用程序作为一个Node.js进程运行，具有Express.js、实时指标 Socket.IO 和内存状态。侧车过程处理延迟探测，以避免在主事件环路上测量开销。

主要组成部分

  • 主要流程 - Express.js 具有实时仪表盘更新 Socket.IO 的服务器
  • 侧车工艺 - 通过 Azure前端探测应用来测量请求延迟的独立进程
  • 工人流程 - CPU 压力和慢速请求仿真使用子进程以避免阻塞主事件环
