📈 Azure Guia de Testes de Carga

Endpoint de teste de carga projetado para integração Azure Testes de Carga

🔬 Visão geral

O Performance Problem Simulator inclui um endpoint dedicado para testes de carga projetado para uso em testes de carga Azure . Esse ponto final simula o realismo comportamento de aplicação que se degrada graciosamente sob carga.

O /api/loadtest O endpoint é projetado para:

  • Realizar trabalho CPU sustentado (ciclos de spin-loop intercalados com intensidade configurável)
  • Mantenha buffers de memória durante toda a duração da solicitação (dividida entre o monte V8 e o buffer nativo para pressão realista)
  • Degrada de forma gradual conforme as solicitações concorrentes aumentam (padrão de limite suave)
  • Faça exceções aleatórias quando requisições concorrentes excedem um limite configurável
  • Eventualmente acionar o tempo de 230 segundos do Azuresob carga extrema

⚙️ Configuração Azure Teste de Carga

  1. No Azure Portal, crie um recurso de Teste de Carga Azure
  2. Crie um novo teste e configure a URL de destino usando uma destas opções:

GET com Parâmetros de Consulta

Use a URL de destino diretamente — sem necessidade de script JMeter:

# With defaults
https://your-app.azurewebsites.net/api/loadtest

# Custom parameters
https://your-app.azurewebsites.net/api/loadtest?workIterations=2000&bufferSizeKb=50000

# Event loop only (no CPU work)
https://your-app.azurewebsites.net/api/loadtest?workIterations=0&bufferSizeKb=100
  1. Configurar o padrão de carga (aumento de carga, regime estacionário, descida)
  2. Configure a integração Azure Monitor para correlacionar métricas

📋 Referência de Parâmetros de Consulta

Cada parâmetro controla um recurso específico de forma independente:

Parâmetro Padrão Recurso Alvo Descrição
workIterations 700 CPU Iterações de spin-loop por ciclo de trabalho CPU . ms por ciclo = workIterations/10. Padrão 700 → 70ms de giro por ciclo.
bufferSizeKb 10000 Memória Memória mantida durante toda a duração da solicitação na base de dados (10 MB padrão). Dividi 50/50 entre o monte V8 e o buffer nativo.
baselineDelayMs 1000 Loop de Eventos Duração mínima da solicitação (ms). CPU trabalho e sono intercalados ao longo do tempo.
softLimit 20 Concorrência Solicitações simultâneas antes do início da degradação. Menor = degradação mais rápida.
degradationFactor 1000 Concorrência Atraso adicional (ms) por requisição sobre softLimit.
errorAboveConcurrent 20 Caos Limiar de solicitação concorrente para injeção de erro. Quando requisições concorrentes excedem esse valor, erros podem ser lançados. Igual ao softLimit por padrão. Defina para 0 e desative.
errorPercent 20 Caos Chance percentual (0-100) de lançar uma exceção aleatória quando acima do limiar de erro Acima do concorrente.

📊 Compreendendo o Limite Suave

O limite suave cria uma curva de degradação onde os tempos de resposta aumentam conforme solicitações concorrentes excedem o limite. A fórmula de degradação é: 

additionalDelay = max(0, concurrent - softLimit) × degradationFactor

Nota: Os tempos reais de resposta variam dependendo do seu Azure App Service nível atual Carga e outros fatores. Use Azure Teste de Carga para estabelecer o desempenho de referência Para o seu ambiente específico. Os parâmetros padrão são ajustados para um trabalhador Premium0V3. Qualquer outro tamanho de SKU ou operador precisará de afinação individual.

Exemplo de Padrão de Degradação

Com as configurações padrão (softLimit=20, degradationFactor=1000):

Solicitações Simultâneas Atraso adicional Tempo de Resposta Esperado
10 0ms ~1000ms (apenas linha base)
20 0ms ~1000ms (no limite suave)
30 10.000ms ~11s
50 30.000ms ~31s

💥 Injeção de Erro

O endpoint de teste de carga pode injetar exceções aleatórias para simular o realismo Falhas de aplicação. Isso é controlado por dois parâmetros:

  • errorAboveConcurrent (padrão: 20) - Limiar de solicitação concorrente antes que erros possam começar (igual ao softLimit)
  • errorPercent (padrão: 20) - Probabilidade de erro uma vez atingido o limiar

💡 Injeção de Erro Baseada em Concorrência: A injeção de erro é ativada por padrão e aciona quando solicitações concorrentes excedem errorAboveConcurrent (padrão: 20, igual a softLimit). Isso significa que erros começam naturalmente a ocorrer quando o sistema começa a se degradar sob carga. Cenário errorAboveConcurrent=0 para desabilitar completamente a injeção de erros.

Tipos de Exceção

Exceções aleatórias são selecionadas de um conjunto de exceções realistas em JavaScript/Node.js:

Tipo de Exceção Mensagem de exemplo
Error ErrorOperacionalInvalid: A operação não é válida devido ao estado atual
TypeError Não é possível ler propriedades de nulo
ReferenceError Referência de objeto não definida para uma instância de um objeto
RangeError O índice estava fora dos limites do array
Error TimeoutError: A operação expirou
Error IOException: Impossível de ler dados da conexão de transporte
Error OutOfMemoryError: Memória insuficiente para continuar a execução

Exemplos de Testes de Caos

# Errors when concurrent requests exceed 10 (50% chance each)
/api/loadtest?errorAboveConcurrent=10&errorPercent=50

# Errors when concurrent requests exceed 25 (rare, 5% chance)
/api/loadtest?errorAboveConcurrent=25&errorPercent=5

# Disable random errors (enabled by default at softLimit)
/api/loadtest?errorAboveConcurrent=0

🔍 Diagnóstico de Resultados de Testes de Carga

1. Application Insights Integração

  • Vá para Application Insights > Apresentação
  • Filtrar por nome da operação GET /api/loadtest
  • Observe percentis de tempo de resposta (p50, p95, p99)
  • Verifique as taxas de falha quando a injeção de erro estiver ativada

2. Azure Monitorar Métricas

  • Pedidos - Total de solicitações e taxa
  • Tempo de Resposta - Latências médias e percentuais
  • Erros do servidor Http - 5xx erros de exceções aleatórias
  • CPU Percentual - Deve aumentar com a carga
  • Conjunto de Trabalho de Memória - Deve aumentar com bufferSizeKb

3. Endpoint de Estatísticas de Teste de Carga

Consulte o endpoint de estatística durante o teste:

GET /api/loadtest/stats

{
  "currentConcurrentRequests": 127,
  "totalRequestsProcessed": 45230,
  "totalExceptionsThrown": 23,
  "averageResponseTimeMs": 847.5
}

📤 Formato de Resposta

Cada solicitação bem-sucedida retorna:

{
  "elapsedMs": 1247,
  "queueTimeMs": 15,
  "workTimeMs": 1232,
  "concurrentRequestsAtStart": 42,
  "degradationDelayAppliedMs": 8500,
  "workIterationsCompleted": 700,
  "memoryAllocatedBytes": 10240000,
  "workCompleted": true,
  "message": "Load test completed in 1247ms"
}

Campos de resposta

Campo Descrição
elapsedMs Duração total da solicitação em milissegundos (incluindo tempo de fila)
queueTimeMs Tempo gasto esperando na fila de loop de eventos antes do início do trabalho
workTimeMs Tempo real gasto trabalhando (elapsedMs - queueTimeMs)
concurrentRequestsAtStart Número de solicitações simultâneas quando essa solicitação começou
degradationDelayAppliedMs Atraso adicional aplicado devido ao excesso do limite suave
workIterationsCompleted Número de ciclos de trabalho CPU concluídos
memoryAllocatedBytes Tamanho do buffer de memória alocado para esta solicitação
workCompleted Se todo o trabalho foi concluído com sucesso
message Mensagem de conclusão legível por humanos

✅ Melhores Práticas

FAZ:

  • ✅ Comece com parâmetros padrão para estabelecer uma linha de base
  • ✅ Monitore métricas Azure durante testes de carga
  • ✅ Use o endpoint /api/loadtest/stats para acompanhar a concorrência
  • ✅ Aumente gradualmente a carga para encontrar pontos de ruptura
  • ✅ Correlacione os resultados do teste de carga com Application Insights
  • ✅ Teste com padrões de carga realistas (ramp up, steady state, ramp down)

NÃO FAÇA:

  • ❌ Comece com valores extremos de parâmetros
  • ❌ Ignore métricas de memória ao usar um grande bufferSizeKb
  • ❌ Desative a injeção de erro (errorAcimaConcurrente=0) se quiser ver falhas realistas
  • ❌ Suponha que os resultados de um SKU se aplicam a outro
