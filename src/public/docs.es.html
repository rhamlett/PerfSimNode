📚 Documentación

Referencia completa de API y guías para herramientas de simulación de rendimiento de PerfSimNode

🎯 Simulador de Problemas de Rendimiento

Una aplicación educativa Azure App Service que crea intencionadamente problemas de rendimiento con fines de aprendizaje y demostración.

Propósito

Esta aplicación está diseñada para ayudar a Azure App Service Ingenieros de Soporte y Asesores Técnicos:

📖
Aprende

Cómo diagnosticar problemas comunes de rendimiento en Azure App Service

🔧
Práctica

Uso de Azure herramientas de diagnóstico y monitorización

🎭
Demostrar

Anti-patrones de rendimiento en un entorno controlado

👥
Tren

Apoyar a los equipos en la identificación y resolución de problemas de rendimiento

⚠️ Advertencia: ¡Esta aplicación crea intencionadamente problemas de rendimiento!

  • 🔥 CPU estrés - Genera procesos hijos que ejecutan operaciones criptográficas para consumir CPU núcleos
  • 📊 Presión de memoria - Asigna y mantiene grandes búferes para aumentar el uso de memoria
  • 🟣 Bloqueo de bucle de eventos - Ejecuta operaciones síncronas que bloquean el hilo principal
  • 🐢 Solicitudes lentas - Genera solicitudes de larga duración para el análisis del perfilador
  • 💥 Caídas de aplicaciones - Activa errores fatales en pruebas Azure monitorización de accidentes
  • ❌ Solicitudes fallidas - Genera HTTP errores 5xx para probar AppLens y Application Insights

Solo despliega esta aplicación en entornos aislados y no de producción.

CPU Simulación de tensiones

Genera un alto uso de CPU utilizando procesos del sistema operativo separados mediante child_process.fork().

Cómo funciona

A diferencia de la quema de CPU ingenua en el hilo principal (que bloquearía Node.js), esta simulación genera procesos hijos separados que se ejecutan cada uno crypto.pbkdf2Sync() en un bucle cerrado. El planificador del sistema operativo distribuye estos procesos entre CPU núcleos.

💡 Node.js Perspectiva:CPUtrabajo intensivo debe aislarse del bucle principal de eventos. Los procesos hijos garantizan una verdadera utilización multinúcleo sin bloquear el manejo de solicitudes.

Uso

POST /api/simulations/cpu
Content-Type: application/json

{
  "intensity": "high",
  "durationSeconds": 30
}

Parámetros

ParámetroValoresDescripción
Intensidad"moderado" | "Alto"CPU nivel de intensidad de estrés
duraciónSegundos1-300Cuánto tiempo ejecutar la simulación

Efectos esperados

  • CPU tile/gráfico aumenta al porcentaje objetivo
  • El retardo del bucle de eventos se mantiene BAJO (el trabajo está en procesos hijos)
  • La latencia de las peticiones puede aumentar ligeramente debido a CPU contención
  • Múltiples node Procesos visibles en top/htop
📊 Sobre CPU métricas: El panel de control CPU muestrea a intervalos de 250 ms usando en bruto os.cpus() datos. Este enfoque de alta resolución puede mostrar fluctuaciones significativas incluso en reposo debido a GC, sondas de rotación y programación de contenedores. Céntrate en el Tendencia durante las simulaciones, no en los picos individuales. Las herramientas de producción (Azure Monitor) muestrean cada minuto con promedia.

Herramientas de diagnóstico

  • top / htop - Ver todos los procesos de los nodos
  • ps aux | grep node - Ver árbol de procesos hijo
  • Azure App Service Diagnósticos → CPU análisis de profundidad
  • Application Insights → Métricas de rendimiento

Simulación de presión de memoria

Asigna y retiene búferes de memoria llenos de datos aleatorios para simular fugas de memoria.

Cómo funciona

Usos Buffer.alloc() para asignar memoria que se mantiene hasta que se libere explícitamente. Los buffers se llenan con datos aleatorios para evitar la optimización V8. Múltiples asignaciones pueden coexistir (comportamiento de apilamiento).

💡 Node.js Perspectiva: El recogedor de basura de V8 utiliza la recogida generacional. Asignaciones grandes pueden causar pausas de GC más largas, visibles como picos de lag en el bucle de eventos.

Uso

# Allocate memory
POST /api/simulations/memory
{"sizeMb": 100}

# Release memory (use the returned ID)
DELETE /api/simulations/memory/{id}

Parámetros

ParámetroDistribuciónDescripción
sizeMb1-500Memoria para asignar en megabytes

Efectos esperados

  • La memoria de heap aumenta según el tamaño de asignación
  • La memoria RSS aumenta proporcionalmente
  • Picos breves de retardo en bucle de eventos durante la asignación (GC)
  • Patrón de dientes de sierra como GC recorridos entre asignaciones

Herramientas de diagnóstico

  • node --inspect - Tomar instantáneas de heap mediante Chrome DevTools
  • Azure App Service → Métricas de memoria
  • Application Insights → customMetrics para estadísticas de heap
  • process.memoryUsage() - Acceso programático

Simulación de bloqueo de bucle de eventos

Demuestra cómo las operaciones síncronas bloquean el bucle de eventos Node.js, haciendo que el servidor sea completamente insensible.

⚠️ Advertencia: Durante esta simulación, el servidor no responderá en absoluto a todas las solicitudes. WebSocket conexiones pueden quedar en suspensión. El salpicadero se congela.

Cómo funciona

Realiza síncronos crypto.pbkdf2Sync() Directamente en el hilo principal. Todas las E/S pendientes, temporizadores y solicitudes entrantes se colocan en cola hasta que se complete el bloqueo.

💡 Diferencia clave con CPU estrés:CPU stress utiliza procesos hijos (el servidor sigue respondiendo). El bloqueo de bucle de eventos se ejecuta en el hilo principal (el servidor deja de responder). ¡Consulta la métrica de Lag del Bucle de Eventos para distinguirlos!

Uso

POST /api/simulations/eventloop
{"durationSeconds": 5}

Síntomas a observar

  • Picos de lag en el bucle de eventos para igualar la duración del bloqueo
  • Todas las solicitudes se colocan y completan juntas tras desbloquear
  • WebSocket los latidos fallan (el panel de control se congela)
  • Los puntos de la sonda se vuelven rojos durante el bloqueo
  • Los chequeos de salud fallan/tiempo muerto

Herramientas de diagnóstico

  • perf_hooks.monitorEventLoopDelay() - Integrado en las métricas de esta app
  • Clinic.js Doctor - Analizar problemas de bucles de eventos
  • 0x - Perfilado de grafos de llama para puntos calientes síncronos

Simulación de peticiones lentas

Simula respuestas de HTTP lentas utilizando múltiples patrones de bloqueo para observar diferentes comportamientos de contención de recursos.

Patrones de bloqueo

PatrónDescripciónRecursos afectados
setTimeoutRetraso no bloqueante (por defecto)Ninguno: el servidor sigue respondiendo
libuvSatura la piscina de hilo libuvOperaciones de FS, DNS y Cripto en cola
workerApariciones bloqueando hilos de trabajadoresCPU, similar a .NET ThreadPool
💡 .NET Paralelismo: El libuv y worker Los patrones simulan la falta de acumulación de hilos, similar a los patrones de bloqueo sync-over-async en .NET que agotan ThreadPool roscas.

Uso

# Non-blocking (default)
GET /api/simulations/slow?delaySeconds=10

# libuv thread pool saturation
GET /api/simulations/slow?delaySeconds=10&blockingPattern=libuv

# Worker thread blocking
GET /api/simulations/slow?delaySeconds=10&blockingPattern=worker

Efectos esperados por patrón

setTimeout (No Bloqueo)

  • El punto final lento toma un tiempo de retardo configurado
  • La latencia de la sonda sigue siendo normal
  • El lag del bucle de eventos se mantiene cerca de cero

Saturación de la piscina de hilo de Libuv

  • Las operaciones del sistema de archivos se ralentizan
  • Las búsquedas DNS se vuelven lentas
  • HTTP solicitudes siguen procesándose con normalidad

Bloqueo de hilo de trabajador

  • CPU aumento del uso
  • Aumenta el número de hilos
  • Bucle del evento principal intacto

Controles del salpicadero

  • Patrón de bloqueo: setTimeout, libuv o worker
  • Retraso: Cuánto tiempo tarda cada petición (1-300 segundos)
  • Intervalo: Tiempo entre solicitudes automatizadas
  • Solicitudes máximas: Cuántos enviar antes de parar

Casos de uso

  • Pruebas de configuraciones de tiempo de espera
  • Simulación de la falta de pool de hilos
  • Comprendiendo el comportamiento de bloqueo frente a no bloqueo

Solicitudes fallidas

Genera HTTP errores de servidor 5xx para probar monitorización de errores, alertas y Azure AppLens diagnósticos.

Cómo funciona

La simulación de Solicitudes Fallidas realiza solicitudes internas de HTTP que están garantizadas para fallar con 500 códigos de estado. Cada solicitud realiza trabajo real (CPU estrés, asignación de memoria, retraso) antes de lanzar una excepción seleccionada aleatoriamente de un conjunto de 17 tipos de error.

Tipos de error

Tipo de errorDescripción
ErrorInválidaOperaciónTIntento de operación inválido
TypeErrorInadaptación de tipo o referencia nula
TimeOutErrorLa operación superó el límite de tiempo
ErrorDeDeMemoriaFueraFallo en la asignación de memoria
ErrorConexiónBase de DatosNo se puede conectar a la base de datos
AutenticationErrorFallo de autenticación
NetworkErrorFallo en la comunicación de red
ServiceUnavailableErrorServicio backend no disponible

Además de 9 tipos de error adicionales para firmas de error diversas.

Controles del salpicadero

  • Recuento de solicitudes: Número de solicitudes fallidas para generar (1-50)

Observando en Azure

AppLens

  • Navegar a Diagnosticar y resolver problemas → HTTP Errores del servidor
  • Ver pico de tasa de error 5xx durante la simulación
  • Examinar la distribución de errores por tipo de excepción

Application Insights

  • Fracasos hoja → pestaña Operaciones
  • Ver tipos de excepción y trazas de pila
  • Analizar patrones de agrupación de errores

Casos de uso

  • Entrenamiento Azure AppLens diagnóstico de errores
  • Pruebas Application Insights seguimiento de fallos
  • Verificación de alertas y notificaciones de errores
  • Practica interpretando diferentes tipos de error

Simulación de accidentes

Intencionadamente bloquea el proceso Node.js para probar la recuperación y observar el comportamiento de reinicio.

⚠️ Advertencia: Estas operaciones terminarán el proceso. En Azure App Service, el proceso se reiniciará automáticamente. El panel de control se desconecta y vuelve a conectar.

Tipos de choque

TipoPunto finalMétodoEfecto
FailFast/crash/failfastprocess.abort()SIGABRT inmediato, descarga de núcleos
Desbordamiento de pila/crash/stackoverflowRecursión infinitaRangeError, pila de llamadas superada
Excepción/crash/exceptionthrow ErrorLa excepción no gestionada termina
OOM/crash/memoryAsignación infinitaAgotamiento absoluto, OOM matar

Comportamiento esperado

  • El proceso termina inmediatamente
  • El panel muestra desconexión
  • Azure App Service proceso de reinicio automático
  • Tras reiniciar: nuevo ID de proceso, simulaciones aprobadas, métricas nuevas
  • El registro de eventos muestra: "🔄¡SOLICITUD REINICIADA! Identificación del proceso modificada..."

Herramientas de diagnóstico

  • Azure App Service → Diagnosticar y Resolver Problemas → Fallos de Aplicación
  • Flujo de registros (registros de fallos en tiempo real)
  • Kudu → LogFiles → Application
  • Application Insights → Hoja de fallos

Configuración opcional

Puedes personalizar el comportamiento de la aplicación usando Azure App Service variables de entorno.

Tasa de Prueba de Salud

Controla con qué frecuencia el servidor envía sondas de salud para medir la latencia de las solicitudes. Todas las sondas se enrutan a través del frontend Azure para capturar latencias realistas de extremo a extremo.

Variable de entorno

Nombre de la variable Descripción Default
HEALTH_PROBE_RATE Intervalo de sonda en milisegundos. Mínimo 100ms. 200

Configuración mediante Azure CLI

# Slow down probes if CLR profiler shows overlapping requests
az webapp config appsettings set --name $APP_NAME --resource-group $RESOURCE_GROUP \
  --settings HEALTH_PROBE_RATE=400

Tiempo de espera en reposo

Cuando la aplicación está inactiva (sin conexiones al panel ni solicitudes de pruebas de carga), las sondas de salud se suspenden automáticamente para reducir el tráfico de red innecesario al frontend de Azurey a la telemetría Application Insights .

Variable de entorno

Nombre de la variable Descripción Default
IDLE_TIMEOUT_MINUTES Minutos de inactividad antes de suspender las sondas de salud. La actividad se reanuda automáticamente cuando se abre el panel de control o se recibe cualquier solicitud. 20

Configuración mediante Azure CLI

# Extend idle timeout to 30 minutes
az webapp config appsettings set --name $APP_NAME --resource-group $RESOURCE_GROUP \
  --settings IDLE_TIMEOUT_MINUTES=30

Pie de página personalizado

Establece un mensaje personalizado en el pie de página que aparezca en todas las páginas. El pie de página soporta HTML, lo que permite incluir enlaces.

Variable de entorno

Nombre de la variable Descripción
PAGE_FOOTER Contenido HTML para la segunda línea del pie de página. Si no está activado, solo se muestra la descripción de la app y la información de la build.

Valor de ejemplo

Created by <a href="https://yoursite.com">Your Team</a> for training purposes

Configuración mediante Azure CLI

# Set the footer via Azure CLI
az webapp config appsettings set --name $APP_NAME --resource-group $RESOURCE_GROUP \
  --settings 'PAGE_FOOTER=Created by <a href="https://yoursite.com">Your Team</a> for training purposes'

Ambientación a través de Azure Portal

  1. Navega hasta tu App Service en el Portal Azure
  2. Ir a Escenarios → Variables de entorno
  3. Clic + Añadir
  4. Nombre del conjunto: PAGE_FOOTER
  5. Valor fijado: Contenido del pie de página HTML
  6. Clic Solicita, entonces Confirmar

GitHub Enlace al Repositorio

Opcionalmente, muestra un enlace GitHub Repositorio en el menú de navegación de la barra lateral. El enlace está oculto por defecto. Cuando se proporcionan ambas variables de entorno, el enlace aparecerá en el menú y apuntará a https://github.com/{GITHUB_USER_NAME}/{GITHUB_REPO_NAME}.

Variables ambientales

Nombre de la variable Descripción
GITHUB_USER_NAME GitHub nombre de usuario o de la organización. Es necesario mostrar el enlace.
GITHUB_REPO_NAME GitHub nombre del repositorio. Es necesario mostrar el enlace.

Configuración mediante Azure CLI

# Set the GitHub link via Azure CLI
az webapp config appsettings set --name $APP_NAME --resource-group $RESOURCE_GROUP \
  --settings GITHUB_USER_NAME=your-username GITHUB_REPO_NAME=your-repo-name

Ambientación a través de Azure Portal

  1. Navega hasta tu App Service en el Portal Azure
  2. Ir a Escenarios → Variables de entorno
  3. Clic + Añadir y nombre del conjunto: GITHUB_USER_NAME, Valor: tu nombre de usuario u organización GitHub
  4. Clic + Añadir y nombre del conjunto: GITHUB_REPO_NAME, Valor: nombre de tu repositorio
  5. Clic Solicita, entonces Confirmar

Selección de idioma

La interfaz del panel de control puede mostrarse en cualquier idioma soportado por Azure Traductor de Servicios Cognitivos. La traducción ocurre una vez al inicio usando el APIde texto traductor.

Cómo funciona

  1. Set UI_LANGUAGE a un código de lenguaje compatible (por ejemplo, es, ja, fr)
  2. Al iniciar, la app traduce las cadenas de la interfaz en inglés y las páginas del documento HTML al idioma de destino
  3. Los archivos traducidos se almacenan en caché en el disco, y los arranques posteriores con el mismo idioma saltan la traducción
  4. Si el archivo original en inglés cambia, la caché se invalida y se vuelve a traducir automáticamente

Variables ambientales

Nombre de la variable Descripción Default
UI_LANGUAGE Código en el idioma destino (por ejemplo, en, es, ja, fr, de, zh-Hans) en
TRANSLATOR_API_KEY Azure Recurso traductor API clave (necesaria para el idioma no inglés) (vacío)
TRANSLATOR_ENDPOINT Azure URL del final del traductor https://api.cognitive.microsofttranslator.com
TRANSLATOR_REGION Azure región del recurso traductor eastus

Configuración en Azure App Service

  1. Crea un recurso de traductor Azure en el Portal Azure
  2. Copia la clave API y la región de los recursos Claves y punto final Hoja
  3. Navega hacia tu App Service → Escenarios → Variables de entorno
  4. Añadir UI_LANGUAGE, TRANSLATOR_API_KEY, y TRANSLATOR_REGION
  5. Clic Solicita, entonces Confirmar — la aplicación se traducirá en el próximo reinicio

Notas

  • Cuando UI_LANGUAGE=en (por defecto), no se produce traducción y no se necesita ningún recurso de traductor
  • La traducción es "traducir una vez al inicio" — no hay carga de traducción en tiempo de ejecución
  • Los términos técnicos (Azure, CPU, API, JSON, etc.) están protegidos contra la traducción mediante una lista de no traducción
  • Los mensajes de registro de eventos del lado del servidor incluyen claves de traducción, por lo que el panel de control las muestra en el idioma seleccionado

Arquitectura

La aplicación se ejecuta como un proceso Node.js con Express.js, Socket.IO para métricas en tiempo real y estado en memoria. Un proceso sidecar gestiona la sondeación de latencia para evitar medir sobrecarga en el bucle principal de eventos.

Componentes clave

  • Proceso principal - Express.js servidor con Socket.IO para actualizaciones en tiempo real del panel
  • Proceso de sidecar- Proceso separado que mide la latencia de la solicitud sondeando la aplicación a través del frontend de Azure
  • Procesos de los trabajadores - CPU simulaciones de estrés y peticiones lentas utilizan procesos hijos para evitar bloquear el bucle principal de eventos
