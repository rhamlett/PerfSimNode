🎯 Performance Problem Simulator

An educational Azure App Service application that intentionally creates performance problems for learning and demonstration purposes.

Purpose

This application is designed to help Azure App Service Support engineers and Technical Advisors:

📖
Learn

How to diagnose common performance issues in Azure App Service

🔧
Practice

Using Azure diagnostics and monitoring tools

🎭
Demonstrate

Performance anti-patterns in a controlled environment

👥
Train

Support teams on identifying and resolving performance problems

⚠️ Warning - This application intentionally creates performance problems!

  • 🔥 CPU stress - Spawns child processes running crypto operations to consume CPU cores
  • 📊 Memory pressure - Allocates and retains large buffers to increase memory usage
  • 🟣 Event loop blocking - Executes synchronous operations that block the main thread
  • 🐢 Slow requests - Generates long-running requests for profiler analysis
  • 💥 Application crashes - Triggers fatal errors for testing Azure Crash Monitoring
  • ❌ Failed requests - Generates HTTP 5xx errors for testing AppLens and Application Insights

Only deploy this application in isolated, non-production environments.

Architecture

The application runs as a Node.js process with Express.js, Socket.IO for real-time metrics, and in-memory state. A sidecar process handles latency probing to avoid measuring overhead on the main event loop.

Key Components

  • Main Process - Express.js server with Socket.IO for real-time dashboard updates
  • Sidecar Process - Separate process that measures request latency by probing the app through Azure's frontend
  • Worker Processes - CPU stress and slow request simulations use child processes to avoid blocking the main event loop

CPU Stress Simulation

Generates high CPU usage using separate OS processes via child_process.fork().

How It Works

Unlike naive CPU burning in the main thread (which would block Node.js), this simulation spawns separate child processes that each run crypto.pbkdf2Sync() in a tight loop. The OS scheduler distributes these processes across CPU cores.

💡 Node.js Insight: CPU-intensive work must be isolated from the main event loop. Child processes guarantee true multi-core utilization without blocking request handling.

Usage

POST /api/simulations/cpu
Content-Type: application/json

{
  "intensity": "high",
  "durationSeconds": 30
}

Parameters

ParameterValuesDescription
intensity"moderate" | "high"CPU stress intensity level
durationSeconds1-300How long to run the simulation

Expected Effects

  • CPU tile/chart increases to target percentage
  • Event loop lag stays LOW (work is in child processes)
  • Request latency may increase slightly due to CPU contention
  • Multiple node processes visible in top/htop
📊 About CPU Metrics: The dashboard samples CPU at 250ms intervals using raw os.cpus() data. This high-resolution approach may show significant fluctuations even at idle due to GC, curl probes, and container scheduling. Focus on the trend during simulations, not individual spikes. Production tools (Azure Monitor) sample every 1 minute with averaging.

Diagnostic Tools

  • top / htop - View all node processes
  • ps aux | grep node - See child process tree
  • Azure App Service Diagnostics → CPU drill-down
  • Application Insights → Performance metrics

Memory Pressure Simulation

Allocates and retains memory buffers filled with random data to simulate memory leaks.

How It Works

Uses Buffer.alloc() to allocate memory that is held until explicitly released. Buffers are filled with random data to prevent V8 optimization. Multiple allocations can coexist (stacking behavior).

💡 Node.js Insight: V8's garbage collector uses generational collection. Large allocations may cause longer GC pauses, visible as event loop lag spikes.

Usage

# Allocate memory
POST /api/simulations/memory
{"sizeMb": 100}

# Release memory (use the returned ID)
DELETE /api/simulations/memory/{id}

Parameters

ParameterRangeDescription
sizeMb1-500Memory to allocate in megabytes

Expected Effects

  • Heap memory increases by allocation size
  • RSS memory increases proportionally
  • Brief event loop lag spikes during allocation (GC)
  • Saw-tooth pattern as GC runs between allocations

Diagnostic Tools

  • node --inspect - Take heap snapshots via Chrome DevTools
  • Azure App Service → Memory metrics
  • Application Insights → customMetrics for heap stats
  • process.memoryUsage() - Programmatic access

Event Loop Blocking Simulation

Demonstrates how synchronous operations block the Node.js event loop, making the server completely unresponsive.

⚠️ Warning: During this simulation, the server will be completely unresponsive to all requests. WebSocket connections may timeout. The dashboard will freeze.

How It Works

Performs synchronous crypto.pbkdf2Sync() directly in the main thread. All pending I/O, timers, and incoming requests are queued until the blocking completes.

💡 Key Difference from CPU Stress: CPU stress uses child processes (server stays responsive). Event loop blocking runs in the main thread (server becomes unresponsive). Check the Event Loop Lag metric to distinguish them!

Usage

POST /api/simulations/eventloop
{"durationSeconds": 5}

Symptoms to Observe

  • Event loop lag spikes to equal the blocking duration
  • All requests queue and complete together after unblock
  • WebSocket heartbeats fail (dashboard freezes)
  • Probe dots turn red during the block
  • Health checks fail/timeout

Diagnostic Tools

  • perf_hooks.monitorEventLoopDelay() - Built into this app's metrics
  • Clinic.js Doctor - Analyze event loop issues
  • 0x - Flame graph profiling for synchronous hotspots

Slow Request Simulation

Simulates slow HTTP responses using multiple blocking patterns to observe different resource contention behaviors.

Blocking Patterns

PatternDescriptionAffected Resources
setTimeoutNon-blocking delay (default)None - server stays responsive
libuvSaturates libuv thread poolfs, dns, crypto operations queue up
workerSpawns blocking worker threadsCPU, similar to .NET ThreadPool
💡 .NET Parallel: The libuv and worker patterns simulate thread pool starvation similar to sync-over-async blocking patterns in .NET that exhaust ThreadPool threads.

Usage

# Non-blocking (default)
GET /api/simulations/slow?delaySeconds=10

# libuv thread pool saturation
GET /api/simulations/slow?delaySeconds=10&blockingPattern=libuv

# Worker thread blocking
GET /api/simulations/slow?delaySeconds=10&blockingPattern=worker

Expected Effects by Pattern

setTimeout (Non-Blocking)

  • Slow endpoint takes configured delay time
  • Probe latency remains normal
  • Event loop lag stays near zero

libuv Thread Pool Saturation

  • File system operations slow down
  • DNS lookups become slow
  • HTTP requests still process normally

Worker Thread Blocking

  • CPU usage increases
  • Thread count increases
  • Main event loop unaffected

Dashboard Controls

  • Blocking Pattern: setTimeout, libuv, or worker
  • Delay: How long each request takes (1-300 seconds)
  • Interval: Time between automated requests
  • Max Requests: How many to send before stopping

Use Cases

  • Testing timeout configurations
  • Simulating thread pool starvation
  • Understanding blocking vs non-blocking behavior
  • Comparing Node.js concurrency model to .NET

Failed Requests

Generate HTTP 5xx server errors for testing error monitoring, alerting, and Azure AppLens diagnostics.

How It Works

The Failed Requests simulation makes internal HTTP requests that are guaranteed to fail with 500 status codes. Each request performs real work (CPU stress, memory allocation, delay) before throwing a randomly selected exception from a pool of 17 error types.

Error Types

Error TypeDescription
InvalidOperationErrorInvalid operation attempted
TypeErrorType mismatch or null reference
TimeoutErrorOperation exceeded time limit
OutOfMemoryErrorMemory allocation failure
DatabaseConnectionErrorUnable to connect to database
AuthenticationErrorAuthentication failure
NetworkErrorNetwork communication failure
ServiceUnavailableErrorBackend service unavailable

Plus 9 additional error types for diverse error signatures.

Dashboard Controls

  • Request Count: Number of failed requests to generate (1-50)

Observing in Azure

AppLens

  • Navigate to Diagnose and solve problems → HTTP Server Errors
  • View 5xx error rate spike during simulation
  • Examine error distribution by exception type

Application Insights

  • Failures blade → Operations tab
  • View exception types and stack traces
  • Analyze error grouping patterns

Use Cases

  • Training Azure AppLens error diagnostics
  • Testing Application Insights failure tracking
  • Verifying error alerting and notifications
  • Practice interpreting diverse error types

Crash Simulation

Intentionally crashes the Node.js process for testing crash recovery and observing restart behavior.

⚠️ Warning: These operations will terminate the process. In Azure App Service, the process will restart automatically. Dashboard will disconnect and reconnect.

Crash Types

TypeEndpointMethodEffect
FailFast/crash/failfastprocess.abort()Immediate SIGABRT, core dump
Stack Overflow/crash/stackoverflowInfinite recursionRangeError, call stack exceeded
Exception/crash/exceptionthrow ErrorUnhandled exception terminates
OOM/crash/memoryInfinite allocationHeap exhaustion, OOM kill

Expected Behavior

  • Process terminates immediately
  • Dashboard shows disconnection
  • Azure App Service auto-restarts process
  • After restart: new Process ID, cleared simulations, fresh metrics
  • Event Log shows: "🔄 APPLICATION RESTARTED! Process ID changed..."

Diagnostic Tools

  • Azure App Service → Diagnose and Solve Problems → Application Crashes
  • Log Stream (real-time crash logs)
  • Kudu → LogFiles → Application
  • Application Insights → Failures blade
