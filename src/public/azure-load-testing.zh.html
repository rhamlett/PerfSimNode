📈 Azure 负载测试指南

为 Azure 负载测试集成设计的负载测试端点

🔬 概述

性能问题模拟器包含专用的负载测试终端 设计用于 Azure 载荷测试。该端点模拟真实感 应用行为在负载下会优雅地下降。

该 /api/loadtest 端点设计目的是：

  • 执行持续的 CPU 功（以可配置强度交错自旋环路）
  • 在整个请求期间保留内存缓冲区（在V8堆和原生缓冲区之间分配以获得真实压力）
  • 随着并发请求增加（软限制模式），性能下降
  • 当并发请求超过可配置阈值时，随机抛出异常
  • 最终在极高负载下触发 Azure230秒的超时

⚙️ 设置 Azure 负载测试

  1. 在 Azure 门户中，创建一个 Azure 负载测试资源
  2. 创建一个新测试，并使用以下选项之一配置目标URL：

GET 查询参数

直接使用目标网址——无需 JMeter 脚本：

# With defaults
https://your-app.azurewebsites.net/api/loadtest

# Custom parameters
https://your-app.azurewebsites.net/api/loadtest?workIterations=2000&bufferSizeKb=50000

# Event loop only (no CPU work)
https://your-app.azurewebsites.net/api/loadtest?workIterations=0&bufferSizeKb=100
  1. 配置负载模式（增加、稳态、下降）
  2. 设置 Azure 监控集成以关联指标

📋 查询参数参考

每个参数独立控制特定资源：

参数 默认 目标资源 描述
workIterations 700 CPU 每个工作周期的旋转循环迭代 CPU 。每周期 ms = 工作迭代次数/10。默认每个周期旋转700→70毫秒。
bufferSizeKb 10000 记忆 内存以KB为单位，整个请求持续时间内保持（默认为10 MB）。在V8堆和原生缓冲区之间各占一半。
baselineDelayMs 1000 事件循环 最小请求时长（毫秒）。 CPU 工作和睡眠交替进行。
softLimit 20 并发 降级开始前需同时请求。较低=降解更快。
degradationFactor 1000 并发 每个请求在softLimit上额外延迟（毫秒）。
errorAboveConcurrent 20 混沌 错误注入的并发请求阈值。当并发请求超过该值时，可能会抛出错误。默认和softLimit一样。设置为0以禁用。
errorPercent 20 混沌 当 errorAboveConcurrent 阈值超过时抛出随机异常的概率（0-100）。

📊 理解软极限

软极限形成一个退化曲线，响应时间随 并发请求超过阈值。降解公式为： 

additionalDelay = max(0, concurrent - softLimit) × degradationFactor

注： 实际响应时间会根据你当前的 Azure App Service 等级而异 负荷，以及其他因素。利用 Azure 负载测试来建立基线性能 针对你具体的环境。默认参数是为 Premium0V3 工作者调校的。 其他的 SKU 或工种尺寸都需要单独调校。

降解模式示例

默认设置下 （softLimit=20, degradationFactor=1000):

并行请求 额外延迟 预期响应时间
10 0毫秒 ~1000毫秒（仅基线）
20 0毫秒 ~1000毫秒（软限制）
30 10,000毫秒 ~11
50 30,000毫秒 ~31秒

💥 错误注入

负载测试端点可以注入随机异常以模拟真实情况 应用程序失败。这由两个参数控制：

  • errorAboveConcurrent （默认：20）- 错误开始前请求并发阈值（与softLimit相同）
  • errorPercent （默认：20）- 达到阈值后出现错误的概率

💡 基于并发的错误注入： 错误注入默认启用，并触发 当并发请求超过时 errorAboveConcurrent （默认：20，与 softLimit). 这意味着当系统在负载下开始退化时，错误自然开始发生。 赛场 errorAboveConcurrent=0 完全禁用错误注入。

例外类型

从一系列真实的JavaScript/Node.js异常中随机选择：

例外类型 示例消息
Error InvalidOperationError：由于当前状态，操作无效
TypeError 无法读取空的属性
ReferenceError 对象引用未设置为对象的实例
RangeError 索引超出了数组范围
Error 超时错误：操作已超时
Error IOException：无法读取传输连接的数据
Error 内存不足错误：内存不足以继续执行

混沌测试示例

# Errors when concurrent requests exceed 10 (50% chance each)
/api/loadtest?errorAboveConcurrent=10&errorPercent=50

# Errors when concurrent requests exceed 25 (rare, 5% chance)
/api/loadtest?errorAboveConcurrent=25&errorPercent=5

# Disable random errors (enabled by default at softLimit)
/api/loadtest?errorAboveConcurrent=0

🔍 负荷测试结果诊断

1. Application Insights 积分

  • 前往 Application Insights >表演
  • 按操作名称进行筛选 GET /api/loadtest
  • 观察响应时间百分位数（第50页，第95页，第99页）
  • 启用错误注入时检查失败率

2. Azure 监测指标

  • 请求 - 总请求数和速率
  • 响应时间 - 平均和百分位延迟
  • HTTP 服务器错误 - 随机异常导致的5xx错误
  • CPU 百分比 - 应随负载增加
  • 记忆工作集 - 应随bufferSizeKb增加

3. 负载测试统计端点

测试期间查询统计端点：

GET /api/loadtest/stats

{
  "currentConcurrentRequests": 127,
  "totalRequestsProcessed": 45230,
  "totalExceptionsThrown": 23,
  "averageResponseTimeMs": 847.5
}

📤 响应格式

每次成功请求返回：

{
  "elapsedMs": 1247,
  "queueTimeMs": 15,
  "workTimeMs": 1232,
  "concurrentRequestsAtStart": 42,
  "degradationDelayAppliedMs": 8500,
  "workIterationsCompleted": 700,
  "memoryAllocatedBytes": 10240000,
  "workCompleted": true,
  "message": "Load test completed in 1247ms"
}

响应场

场地 描述
elapsedMs 总请求时长（以毫秒计）（包括队列时间）
queueTimeMs 在事件循环队列中等待工作开始的时间
workTimeMs 实际工作时间（elapsedMs - queueTimeMs）
concurrentRequestsAtStart 请求开始时并发请求数量
degradationDelayAppliedMs 因超额而追加延迟
workIterationsCompleted 完成的 CPU 工作周期数量
memoryAllocatedBytes 为本次请求分配的内存缓冲区大小
workCompleted 所有工作是否成功完成
message 人类可读的完成信息

✅ 最佳实践

DO：

  • ✅ 从默认参数开始建立基线
  • ✅ 在负载测试中监控 Azure 指标
  • ✅ 用 /api/loadtest/stats 端点来跟踪并发
  • ✅ 逐渐增加负载以找到断裂点
  • ✅ 将载荷测试结果与Application Insights相关
  • ✅ 用真实的负载模式测试（加量、稳态、减速）

不要这样做：

  • ❌ 从极端参数开始
  • ❌ 使用大缓冲区大小Kb时忽略内存度量
  • ❌ 如果你想看到真实的失败，可以禁用错误注入（errorAboveConcurrent=0）
  • ❌ 假设一个 SKU 的结果适用于另一个
