Referência completa de API e guias para ferramentas de simulação de desempenho do PerfSimNode

🎯 Simulador de Problemas de Desempenho

Uma aplicação Azure App Service educacional que intencionalmente cria problemas de desempenho para fins de aprendizagem e demonstração.

Propósito

Esta aplicação foi projetada para ajudar Azure App Service Engenheiros de Suporte e Consultores Técnicos:

📖
Aprenda

Como diagnosticar problemas comuns de desempenho em Azure App Service

🔧
Prática

Usando Azure ferramentas de diagnóstico e monitoramento

🎭
Demonstrar

Anti-padrões de desempenho em um ambiente controlado

👥
Trem

Apoiar as equipes na identificação e resolução de problemas de desempenho

⚠️ Aviso - Este aplicativo cria intencionalmente problemas de desempenho!

  • 🔥 CPU estresse - Spawns processos filhos executando operações criptográficas para consumir CPU núcleos
  • 📊 Pressão de memória - Aloca e mantém grandes buffers para aumentar o uso de memória
  • 🟣 Bloqueio de loop de eventos - Executa operações síncronas que bloqueiam a thread principal
  • 🐢 Pedidos lentos - Gera solicitações de longa duração para análise de perfiladores
  • 💥 A aplicação trava - Aciona erros fatais para testes Azure Monitoramento de Acidentes
  • ❌ Pedidos fracassados - Gera HTTP erros 5xx para testes de AppLens e Application Insights

Implante essa aplicação apenas em ambientes isolados e não produtivos.

CPU Simulação de Estresse

Gera alto uso de CPU usando processos do sistema operacional separados via child_process.fork().

Como Funciona

Diferente da queima de CPU ingênua na thread principal (que bloquearia Node.js), essa simulação gera processos filhos separados que executam crypto.pbkdf2Sync() em um ciclo apertado. O escalonador do sistema operacional distribui esses processos entre CPU núcleos.

💡 Node.js Insight:Trabalho CPU- intensivo deve ser isolado do ciclo principal de eventos. Processos filhos garantem verdadeira utilização multi-core sem bloquear o tratamento de requisições.

Uso

POST /api/simulations/cpu
Content-Type: application/json

{
  "intensity": "high",
  "durationSeconds": 30
}

Parâmetros

ParâmetroValoresDescrição
Intensidade"moderado" | "Alto"CPU nível de intensidade do estresse
duraçãoSegundos1-300Quanto tempo para rodar a simulação

Efeitos Esperados

  • CPU tile/gráfico aumenta em relação ao percentual alvo
  • O atraso do loop de eventos permanece BAIXO (o trabalho está em processos filhos)
  • A latência de requisições pode aumentar ligeiramente devido à CPU contenção
  • Múltiplos node Processos visíveis em top/htop
📊 Sobre CPU Métricas: O painel amostra CPU em intervalos de 250ms usando o raw os.cpus() dados. Essa abordagem de alta resolução pode apresentar flutuações significativas mesmo em repouso devido a GC, sondas de rotação e escalonamento de contêineres. Foque no Tendência Durante simulações, não em picos individuais. Ferramentas de produção (Azure Monitor) amostram a cada 1 minuto com média.

Ferramentas de Diagnóstico

  • top / htop - Visualizar todos os processos de nós
  • ps aux | grep node - Veja árvore de processos filhos
  • Azure App Service Diagnósticos → CPU análise detalhada
  • Application Insights → Métricas de desempenho

Simulação de Pressão de Memória

Aloca e mantém buffers de memória preenchidos com dados aleatórios para simular vazamentos de memória.

Como Funciona

Usos Buffer.alloc() para alocar memória que é mantida até ser liberada explicitamente. Buffers são preenchidos com dados aleatórios para evitar a otimização V8. Múltiplas alocações podem coexistir (comportamento de empilhamento).

💡 Node.js Insight: O coletor de lixo do V8 usa coleta geracional. Grandes alocações podem causar pausas de GC mais longas, visíveis como picos de atraso no loop de eventos.

Uso

# Allocate memory
POST /api/simulations/memory
{"sizeMb": 100}

# Release memory (use the returned ID)
DELETE /api/simulations/memory/{id}

Parâmetros

ParâmetroDistribuiçãoDescrição
sizeMb1-500Memória a ser alocada em megabytes

Efeitos Esperados

  • A memória heap aumenta conforme o tamanho da alocação
  • A memória RSS aumenta proporcionalmente
  • Picos breves de atraso no loop de eventos durante a alocação (GC)
  • Padrão de dentes de serra como GC entre alocações

Ferramentas de Diagnóstico

  • node --inspect - Fazer snapshots de heap via Chrome DevTools
  • Azure App Service → Métricas de memória
  • Application Insights → customMetrics para estatísticas de heap
  • process.memoryUsage() - Acesso programático

Simulação de Bloqueio de Loop de Eventos

Demonstra como operações síncronas bloqueiam o ciclo de eventos Node.js, tornando o servidor completamente inresponsivo.

⚠️ Aviso: Durante essa simulação, o servidor ficará completamente indisponível para responder a todas as solicitações. WebSocket conexões podem expirar. O painel trava.

Como Funciona

Executa síncronos crypto.pbkdf2Sync() diretamente no tópico principal. Todas as E/S pendentes, temporizadores e pedidos recebidos ficam em fila até que o bloqueio seja concluído.

💡 Diferença Chave em relação ao Estresse CPU :CPU stress usa processos filhos (servidor permanece responsivo). O bloqueio de loop de eventos roda na thread principal (o servidor fica sem resposta). Confira a métrica de Event Loop Lag para distingui-los!

Uso

POST /api/simulations/eventloop
{"durationSeconds": 5}

Sintomas a Observar

  • O lag do loop de eventos aumenta para igualar a duração do bloqueio
  • Todas as solicitações entram em fila e se completam juntas após o desbloqueio
  • WebSocket batimentos cardíacos falham (painel trava)
  • Os pontos de sondagem ficam vermelhos durante o bloqueio
  • Exames de saúde falham/tempo de espera

Ferramentas de Diagnóstico

  • perf_hooks.monitorEventLoopDelay() - Incorporado às métricas deste app
  • Clinic.js Doctor - Analisar questões de loops de eventos
  • 0x - Perfil de grafo de chama para hotspots síncronos

Simulação de Solicitação Lenta

Simula respostas lentas de HTTP usando múltiplos padrões de bloqueio para observar diferentes comportamentos de contenção de recursos.

Padrões de bloqueio

PadrãoDescriçãoRecursos Afetados
setTimeoutAtraso não bloqueante (padrão)Nenhum - o servidor continua responsivo
libuvSatura o pool de threads libuvFS, DNS, Operações de Criptomoedas em Fila
workerSpawns bloqueando threads de trabalhadoresCPU, semelhante ao .NET ThreadPool
💡 .NET Paralelo: O libuv e worker Padrões simulam a falta de poço de fios, semelhante a padrões de bloqueio sync-over-async em .NET que esgotam ThreadPool rosca.

Uso

# Non-blocking (default)
GET /api/simulations/slow?delaySeconds=10

# libuv thread pool saturation
GET /api/simulations/slow?delaySeconds=10&blockingPattern=libuv

# Worker thread blocking
GET /api/simulations/slow?delaySeconds=10&blockingPattern=worker

Efeitos Esperados por Padrão

setTimeout (Sem Bloqueio)

  • Endpoint lento leva tempo de atraso configurado
  • A latência da sonda permanece normal
  • O atraso do loop de eventos permanece próximo de zero

Saturação da Piscina de Fios de Libuv

  • As operações do sistema de arquivos desaceleram
  • As buscas DNS ficam lentas
  • HTTP pedidos ainda são processados normalmente

Bloqueio de Fios de Trabalhadores

  • CPU aumento do uso
  • Aumento da contagem de fios
  • Loop do evento principal não afetado

Controles do Painel

  • Padrão de Bloqueio: setTimeout, libuv ou trabalhador
  • Atraso: Quanto tempo cada solicitação leva (1-300 segundos)
  • Intervalo: Tempo entre as solicitações automatizadas
  • Pedidos Máximos: Quantos enviar antes de parar

Casos de Uso

  • Testando configurações de timeout
  • Simulando a falta de pool de threads
  • Entendendo comportamento de bloqueio versus não bloqueio

Pedidos Fracassados

Gerar HTTP erros de servidor 5xx para testar monitoramento de erros, alertas e diagnósticos Azure AppLens .

Como Funciona

A simulação de Solicitações Falhadas faz requisições internas de HTTP que são garantidas para falhar com 500 códigos de status. Cada solicitação realiza trabalho real (CPU estresse, alocação de memória, atraso) antes de lançar uma exceção selecionada aleatoriamente de um conjunto de 17 tipos de erro.

Tipos de erro

Tipo de erroDescrição
ErrorOperacionalDeInvalidOperação inválida tentada
TypeErrorIncompatibilidade de tipo ou referência nula
TimeOutErrorOperação excedeu o limite de tempo
ErroDeSMemóriaForaFalha de alocação de memória
ErroConexãoBase de DadosNão é possível conectar ao banco de dados
AutenticationErrorFalha de autenticação
NetworkErrorFalha na comunicação de rede
ServiceUnavailableErrorServiço backend indisponível

Além de 9 tipos de erro adicionais para assinaturas de erro diversas.

Controles do Painel

  • Contagem de Solicitações: Número de requisições falhadas a serem geradas (1-50)

Observando em Azure

AppLens

  • Navegar para Diagnosticar e resolver problemas → HTTP Erros de servidor
  • Veja o pico de taxa de erro 5xx durante a simulação
  • Examinar a distribuição de erro por tipo de exceção

Application Insights

  • Falhas lâmina → aba Operações
  • Visualizar tipos de exceção e trilhas de pilha
  • Analisar padrões de agrupamento de erros

Casos de Uso

  • Treinamento Azure AppLens diagnóstico de erros
  • Testes Application Insights acompanhamento de falhas
  • Verificação de alertas e notificações de erro
  • Pratique interpretar diferentes tipos de erro

Simulação de Colisão

Intencionalmente trava o processo Node.js para testar a recuperação de falhas e observar o comportamento de reinicialização.

⚠️ Aviso: Essas operações encerrarão o processo. Em Azure App Service, o processo será reiniciado automaticamente. O painel desconecta e reconecta.

Tipos de Colisão

TipoEndpointMétodoEfeito
FailFast/crash/failfastprocess.abort()SIGABRT imediato, despejo do núcleo
Excesso de Pilha/crash/stackoverflowRecursão infinitaErroRange, pilha de chamadas ultrapassada
Exceção/crash/exceptionthrow ErrorExceção não resolvida termina
OOM/crash/memoryAlocação infinitaExaustão acumulada, OOM matar

Comportamento Esperado

  • O processo termina imediatamente
  • O painel mostra desconexão
  • Azure App Service processo de reinicialização automática
  • Após o reinício: novo ID de processo, simulações limpas, métricas novas
  • O Registro de Eventos mostra: "🔄 INSCRIÇÃO REINICIADA! ID do processo alterado..."

Ferramentas de Diagnóstico

  • Azure App Service → Diagnosticar e Resolver Problemas → Falhas de Aplicação
  • Fluxo de Logs (logs de falhas em tempo real)
  • Kudu → LogFiles → Application
  • Application Insights → Lâmina de falhas

Configuração Opcional

Você pode personalizar o comportamento da aplicação usando Azure App Service variáveis de ambiente.

Taxa de Sondagem de Saúde

Controle com que frequência o servidor envia sondas de saúde para medir a latência das requisições. Todas as sondas são roteadas pelo frontend Azure para capturar latência realista de ponta a ponta.

Variável Ambiente

Nome da variável Descrição Padrão
HEALTH_PROBE_RATE Intervalo de sonda em milissegundos. Mínimo 100ms. 200

Configuração via Azure CLI

# Slow down probes if CLR profiler shows overlapping requests
az webapp config appsettings set --name $APP_NAME --resource-group $RESOURCE_GROUP \
  --settings HEALTH_PROBE_RATE=400

Tempo de inatividade

Quando a aplicação está ociosa (sem conexões no painel ou requisições de teste de carga), as sondas de saúde são automaticamente suspensas para reduzir o tráfego desnecessário de rede para a interface e a telemetria Application Insights da Azure.

Variável Ambiente

Nome da variável Descrição Padrão
IDLE_TIMEOUT_MINUTES Minutos de inatividade antes de suspender as sondas de saúde. A atividade é retomada automaticamente quando o painel é aberto ou qualquer solicitação é recebida. 20

Configuração via Azure CLI

# Extend idle timeout to 30 minutes
az webapp config appsettings set --name $APP_NAME --resource-group $RESOURCE_GROUP \
  --settings IDLE_TIMEOUT_MINUTES=30

Rodapé de página personalizado

Defina uma mensagem personalizada no rodapé que apareça em todas as páginas. O rodapé suporta HTML, permitindo incluir links.

Variável Ambiente

Nome da variável Descrição
PAGE_FOOTER Conteúdo HTML para a segunda linha do rodapé. Se não estiver definido, apenas a descrição do app e as informações da build são mostradas.

Exemplo de Valor

Created by <a href="https://yoursite.com">Your Team</a> for training purposes

Configuração via Azure CLI

# Set the footer via Azure CLI
az webapp config appsettings set --name $APP_NAME --resource-group $RESOURCE_GROUP \
  --settings 'PAGE_FOOTER=Created by <a href="https://yoursite.com">Your Team</a> for training purposes'

Ambientação via Azure Portal

  1. Navegue até sua App Service no Portal Azure
  2. Vá para Configurações → Variáveis do ambiente
  3. Click + Adicionar
  4. Nome do Conjunto: PAGE_FOOTER
  5. Valor de Definição: Seu conteúdo do rodapé HTML
  6. Click Candidate-se, então Confirme

GitHub Link do Repositório

Opcionalmente, exiba um link GitHub Repositório no menu de navegação da barra lateral. O link está oculto por padrão. Quando ambas as variáveis de ambiente forem fornecidas, o link aparecerá no menu e apontará para https://github.com/{GITHUB_USER_NAME}/{GITHUB_REPO_NAME}.

Variáveis Ambientais

Nome da variável Descrição
GITHUB_USER_NAME GitHub nome de usuário ou da organização. Obrigatório mostrar o link.
GITHUB_REPO_NAME GitHub nome do repositório. Obrigatório mostrar o link.

Configuração via Azure CLI

# Set the GitHub link via Azure CLI
az webapp config appsettings set --name $APP_NAME --resource-group $RESOURCE_GROUP \
  --settings GITHUB_USER_NAME=your-username GITHUB_REPO_NAME=your-repo-name

Ambientação via Azure Portal

  1. Navegue até sua App Service no Portal Azure
  2. Vá para Configurações → Variáveis do ambiente
  3. Click + Adicionar e nome do conjunto: GITHUB_USER_NAME, Valor: seu nome de usuário GitHub ou organização
  4. Click + Adicionar e nome do conjunto: GITHUB_REPO_NAME, Valor: nome do seu repositório
  5. Click Candidate-se, então Confirme

Seleção de Língua

A interface do painel pode ser exibida em qualquer idioma suportado por Azure Tradutor de Serviços Cognitivos. A tradução ocorre uma vez na inicialização usando o Tradutor de Texto API.

Como Funciona

  1. Cenário UI_LANGUAGE para um código de linguagem suportado (por exemplo, es, ja, fr)
  2. Ao iniciar, o app traduz as strings de interface em inglês e as páginas do documento HTML para o idioma de destino
  3. Arquivos traduzidos são armazenados em cache no disco — startups subsequentes com o mesmo idioma pulam a tradução
  4. Se o arquivo original em inglês for alterado, o cache é invalidado e retraduzido automaticamente

Variáveis Ambientais

Nome da variável Descrição Padrão
UI_LANGUAGE Código no idioma alvo (por exemplo, en, es, ja, fr, de, zh-Hans) en
TRANSLATOR_API_KEY Azure Chave de recurso API tradutor (necessária para línguas não inglesas) (vazio)
TRANSLATOR_ENDPOINT Azure URL do endpoint do tradutor https://api.cognitive.microsofttranslator.com
TRANSLATOR_REGION Azure região do recurso Tradutor eastus

Configuração no Azure App Service

  1. Crie um recurso Azure Tradutor no Portal Azure
  2. Copie a chave API e a região a partir do recurso Chaves e Endpoint Lâmina
  3. Navegue até o seu App Service → Configurações → Variáveis do ambiente
  4. Adicionar UI_LANGUAGE, TRANSLATOR_API_KEY, e TRANSLATOR_REGION
  5. Click Candidate-se, então Confirme — o aplicativo traduzirá na próxima reinicialização

Notas

  • Quando UI_LANGUAGE=en (padrão), não ocorre tradução e nenhum recurso tradutor é necessário
  • Tradução é "traduzir uma vez na inicialização" — não há overhead de tradução em tempo de execução
  • Termos técnicos (Azure, CPU, API, JSON, etc.) são protegidos contra tradução por meio de uma lista de não-tradução
  • Mensagens de log de eventos do lado do servidor incluem chaves de tradução, então o painel as exibe no idioma selecionado

Arquitetura

A aplicação roda como um processo Node.js com Express.js, Socket.IO para métricas em tempo real e estado em memória. Um processo sidecar gerencia a sondagem de latência para evitar medir sobrecarga no loop principal do evento.

Componentes Principais

  • Processo Principal - Express.js servidor com Socket.IO para atualizações em tempo real do painel
  • Processo de Sidecar- Processo separado que mede a latência do pedido sondando o app através da interface de Azure
  • Processos de Trabalhadores - CPU simulações de estresse e requisições lentas usam processos filhos para evitar bloquear o loop principal de eventos
