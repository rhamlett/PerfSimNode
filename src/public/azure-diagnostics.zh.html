☁️ Azure 诊断指南

如何使用 Azure 工具诊断Node.js性能问题，并理解这些指标的含义

📈 理解仪表盘指标

PerfSimNode仪表盘显示实时指标，帮助识别性能问题。以下是每个指标的含义：

⚡ CPU 使用率（%）

衡量内容： Node.js过程所占时间 CPU 百分比。

为什么重要： 高 CPU 表示计算受限操作。在Node.js中，这通常意味着同步代码正在运行（加密、 JSON 解析大型对象、复杂计算）。

正常范围： 怠速时0-30%，请求时的峰值是正常的。

警示信号： 持续>70%表示存在问题。

📊 内存工作集（MB）

衡量内容： Node.js进程已分配并正在积极使用的总内存。

为什么重要： 内存增长但未释放，说明存在泄漏。Node.js使用V8的垃圾回收器，但参考值保持时间过长会导致清理困难。

正常范围： 这取决于应用，但预热后应该会稳定下来。

警示信号： 持续生长（锯齿形纹理可以）。

🧵 事件循环延迟（MS）

衡量内容： 从调回调到实际执行之间的时间。

为什么重要： 这是 sync-over-async 问题的关键指标！ 当同步代码阻断事件循环时，所有待处理的回调都必须等待——包括收到的 HTTP 请求。

正常范围： <10毫秒

警示信号： >100毫秒会导致明显延迟;>1000毫秒意味着严重阻塞。

⏱️ 请求延迟（ms）

衡量内容： 从请求收到到响应发送的时间，包括在事件循环队列中等待的时间。

为什么重要： 直接衡量用户体验。高延迟但低 CPU 通常意味着阻塞或外部依赖问题。

正常范围： 简单请求<150ms。

警示信号： >1 退化;>30多岁，危急。

💡 关键洞察：事件循环延迟Node.js 使用 单线程事件循环.当你看到高事件循环延迟时，说明主线程被阻塞，所有待处理回调都必须等待——这就是为什么一次阻塞操作可以同时影响所有请求。

🔍 诊断场景

以下是诊断该模拟器产生的每种性能问题的方法：

🔥 高 CPU 使用率

症状：CPU高钉顶的指标，请求仍可能正常完成（事件循环未被阻塞）。

发生了什么

  • 仿真通过以下方式生成子进程 fork()
  • 每个子节点运行 CPU密集型加密（PBKDF2）
  • 工作与主事件循环隔离——服务器保持响应
  • 多重 node 可见的过程 top

关键见解

CPU ≠事件循环阻塞的压力！ 子进程的高 CPU 不会阻挡服务器。注意事件循环延迟——即使 CPU很高，延迟依然很低。

Azure 诊断工具

  • App Service 诊断 → CPU 钻探
  • Application Insights → 性能
  • Kudu SSH → top, ps aux | grep node
  • V8 轮郭分析 → --prof 识别热函数

注意点

  • 哪些过程消耗 CPU
  • 父母/子女过程关系
  • 事件循环延迟应保持低
  • V8 配置文件输出显示高刻函数
📊 记忆压力

症状： 记忆会增长，保持高峰，最终可能会崩溃 OOM。

发生了什么

  • 仿真分配缓冲区并保留引用
  • V8 垃圾回收器无法释放持有的内存
  • 模拟缓存失误导致的内存泄漏

Azure 诊断工具

  • App Service 诊断 → 记忆分析
  • Application Insights → 指标
  • 库杜→ 进程探索器
  • V8 轮郭分析 → 通过 Chrome DevTools 进行堆快照

注意点

  • 堆快照用于查找保留对象（使用保留者视图）
  • 比较时间内的快照，以识别不断增长的配置
  • 内存增长模式（泄漏与正常使用）
  • 与特定操作的相关性
🧵 事件环路阻塞

症状： 事件环路延迟激增，所有请求队列，延迟图表显示实时持续高延迟。

发生了什么

  • 同步加密在主线程中以重复的分块形式运行
  • 在区块之间，短暂的让步可以让排队的探针数据刷新
  • 事件循环被阻塞的概率~97%——实际上是无响应的
  • 延迟图表实时更新，显示阻挡效应
  • 与 CPU 压力的主要区别： 服务器无响应！

如何识别

检查事件循环延迟！ 如果它等于区块持续时间，事件循环就会被阻塞。 CPU 应力显示 CPU 高但事件环延迟很低。

Azure 诊断工具

  • Application Insights → 性能
  • App Service 诊断 → 网页应用缓慢
  • Azure 监控→ 指标 用于请求延迟尖峰
  • V8 轮郭分析 → CPU 配置文件在分区显示阻塞代码时

注意点

  • 事件循环延迟 = 块持续时间（~200毫秒默认）
  • 所有请求同时受影响
  • 延迟图表显示模拟过程中持续较高的延迟
  • CPU 可能会高或低（取决于定型作业）
  • V8 配置文件识别同步功能（例如， pbkdf2Sync)
🐢 慢速请求

症状： 具体端点耗时较长，但其他端点是正常的。

发生了什么

  • 请求处理程序的用途 setTimeout() 延误
  • 非阻塞：事件循环继续处理
  • 模拟慢速数据库、外部 API 调用
  • 与事件循环阻断的主要区别： 只有慢端受影响！

如何识别

检查其他请求是否受到影响。 慢请求：只有慢端点才慢。事件循环阻塞：一切都很慢。

Azure 诊断工具

  • Application Insights → 端到端交易
  • Application Insights → 依赖追踪
  • 按终端划分的日志分析→请求时长

注意点

  • 具体哪个端点比较慢？
  • 依赖时序分解
  • 事件环路延迟保持低
💥 应用崩溃

症状： 应用变得无响应、重启或返回502/503错误。

发生了什么

  • 未处理的异常会导致进程崩溃
  • OOM 杀手终止了流程
  • App Service 可能会自动重启

Azure 诊断工具

  • App Service 诊断 → 网页应用重启
  • Application Insights → 故障
  • 库杜→ 日志文件
  • V8 轮郭分析 → 崩溃前的堆快照以识别原因

注意点

  • 崩溃倾倒分析
  • 日志中的异常栈跟踪
  • 崩溃前的内存使用情况（堆快照比较）

🔗 仿真ID相关性

每个仿真实例都被分配了唯一的仿真ID（GUID），实现仪表盘UI与 Azure 诊断之间的端到端关联。

寻找仿真ID

在仪表盘中： 将鼠标悬停在事件日志中任何与仿真相关的消息（带虚线下划线）上，即可查看完整的仿真ID。

Azure： 相同的仿真ID也出现在服务器端日志中，并可在以下格式中找到：

  • Application Insights - 通过仿真ID搜索跟踪和自定义事件
  • 日志分析 - 使用ID作为过滤器的查询日志
  • 容器/App Service 日志- 按 标准/标准GUID 过滤

故障排除工作流程

  1. 从仪表盘开始模拟
  2. 将鼠标悬停在事件日志中的开始消息上，复制模拟ID
  3. 打开 Azure 门户→ Application Insights →日志（或日志分析）
  4. 使用仿真ID查询日志，查看所有相关的服务器端活动
  5. 将仪表盘事件与服务器指标的时间戳关联起来
💡 提示

当多个并发模拟运行或仪表盘被多个用户使用时，模拟ID有助于隔离遥测数据。

🛠️ Azure 诊断工具

诊断Node.js性能问题的关键 Azure 工具：

App Service 诊断

Azure门户内置诊断报告。通过“诊断和解决问题”访问。

最佳用途： 快速健康回顾、 CPU/记忆分析、事故调查。

Application Insights

完整的APM解决方案，支持请求追踪、依赖跟踪和自定义指标。

最佳用途： 端到端追踪、性能趋势、故障分析。

Kudu（SCM）

开发者控制台 yourapp.scm.azurewebsites.net

最佳用途： 实时调试、日志访问、进程管理。

日志分析

基于查询的日志分析，采用KQL技术。

最佳用途： 历史分析、自定义查询、提醒。

Azure 监测指标

平台的指标包括 CPU、内存、请求等。

最佳用途： 实时监控、仪表盘、警报。

崩溃倾倒分析

事故调查时可选择自动或手动排放收集。

最佳用途： 尸检分析，记忆泄漏调查。

🔄 诊断工作流程

在调查Node.js性能问题时，请遵循以下工作流程：

客户报告 → 检查指标 → 识别模式 → 深度探究 → 根本原因

第一步：初步评估

  1. 查看 App Service 健康状况概览
  2. 查看Azure监视器中的 CPU、内存、请求指标
  3. 请注意问题的时间框架

第二步：识别模式

如果你看到...... 可能的问题 下一步
高 CPU，正常响应时间 CPU-bound 处理（预期或错误处理） CPU 通过 Kudu 进行画像分析
CPU高，反应缓慢 CPU 饥饿影响请求 识别热代码路径
正常 CPU，慢，所有反应 事件循环阻断（sync-over-async） 检查事件循环指标，代码审查
正常 CPU，但有些回复很慢 外部依赖延迟 Application Insights 依赖视图
记忆随时间增长 内存泄漏 堆快照分析
突然重启，502个错误 崩溃（例外或 OOM） 崩溃转储，错误日志

步骤3：按问题深入探索工具

🔥 高 CPU
  1. Kudu Process Explorer → Identify process
  2. App Service 诊断→ CPU 深入分析
  3. 简介 node --prof 或 --inspect
🧵 事件环路阻塞
  1. 检查 Application Insights 是否有相关的慢速请求
  2. 在代码中寻找同步操作（如 fs.readFileSync 等）。
  3. 添加监控 perf_hooks.monitorEventLoopDelay()
📊 内存泄漏
  1. 通过Kudu或远程调试进行堆快照
  2. 比较时间上的快照
  3. 注意物体数量增加

🔬 V8 轮郭分析

V8 是驱动 Node.js 的 JavaScript 引擎。它内置了分析工具，帮助识别性能瓶颈、内存泄漏以及 CPU密集的代码路径。

什么是V8画像？

V8 配置分析捕捉了关于你应用运行时行为的详细信息：

  • CPU 画像 — 记录哪些函数消耗最多 CPU 时间，有助于识别热代码路径和优化机会
  • 堆快照 — 捕捉某一时间点的内存状态，显示所有分配的对象及其引用
  • 堆时间线 — 跟踪内存分配以识别泄漏和过度垃圾回收

方法1：命令行分析（--prof）

该 --prof flag 生成详细的 CPU 配置文件，显示每个函数所花费的时间。

步骤1：启用画像运行

# Start your app with profiling
node --prof app.js

# Reproduce the performance issue, then stop the app (Ctrl+C)
# This creates an isolate-*.log file in the current directory

步骤2：处理个人资料

# Convert the V8 log to human-readable format
node --prof-process isolate-0x*.log > profile.txt

第三步：阅读结果

处理后的配置文件显示如下明细：

[Summary]:
   ticks  total  nonlib   name
   5234   75.2%   78.1%  JavaScript
   1023   14.7%   15.3%  C++
    445    6.4%    6.6%  GC
    258    3.7%          Shared libraries

[JavaScript]:
   ticks  total  nonlib   name
   3102   44.6%   46.3%  pbkdf2Sync
    892   12.8%   13.3%  JSON.parse
    ...

寻找刻数高的函数——这些是你 CPU 的热点。

方法二：Chrome DevTools （--检查）

对于带有可视化界面的交互式分析，可以使用Chrome DevTools检查器。

第一步：从检查员开始

# Local development
node --inspect app.js

# Allow remote connections (e.g., from Azure)
node --inspect=0.0.0.0:9229 app.js

步骤2：连接Chrome DevTools

  1. 打开Chrome，导航到 chrome://inspect
  2. 点击“配置......”并加 localhost:9229 （或者你的远程主机）
  3. 你的Node.js过程会出现在“远程目标”下——点击“检查”

第三步：创建 CPU 档案

  1. 去 画像师 标签页（或性能标签页）
  2. 咔哒 开始
  3. 重现性能问题
  4. 咔哒 停下
  5. 分析火焰图——宽条表示慢速

步骤4：拍摄堆快照

  1. 去 记忆 标签页
  2. 选择“堆快照”并点击 拍张快照
  3. 请浏览“摘要”视图，查看对象数量和大小
  4. 比较多个快照以识别内存泄漏

方法3：程序化堆快照

从代码内部或信号生成堆快照：

// In your application code
const v8 = require('v8');
const fs = require('fs');

function takeHeapSnapshot() {
  const filename = `heap-${Date.now()}.heapsnapshot`;
  const stream = fs.createWriteStream(filename);
  v8.writeHeapSnapshot(filename);
  console.log(`Heap snapshot written to ${filename}`);
}

// Trigger on SIGUSR2 signal
process.on('SIGUSR2', takeHeapSnapshot);

然后触发： kill -USR2 <pid>

读取堆快照

开门 .heapsnapshot Chrome DevTools 中的文件：

  1. Open Chrome DevTools（F12）
  2. 前往 记忆 标签页
  3. 咔哒 负载 并选择你的 .heapsnapshot 文件

分析重点观点：

  • 摘要 — 按构造体名称分组的对象。注意意外的数量。
  • 比较 —— 比较两个快照以查看它们之间分配了多少。
  • 遏制 —— 显示对象层级（哪些物体持有对哪些的引用）。
  • 侍从 ——对于任何对象，显示为什么它仍然存在内存（指向 GC 根的参考链）。

Azure- 具体考虑

  • 库杜SSH —— 直接通过以下方式执行分析命令 yourapp.scm.azurewebsites.net
  • 端口转发 —— 远程 --inspect你需要通过CLI或VS Code配置端口转发 Azure
  • 文件下载 —— 使用 Kudu 的文件浏览器 或 az webapp log download 检索配置文件文件
  • 制作影响 — 画像会增加开销;在制作中短暂用于分级或捕获
💡 快速参考
  • CPU 慢？ →用途 --prof 或用Profiler标签查找热函数
  • 记忆力增长？ → 拍摄堆快照并随时间进行比较
  • 事件循环被阻断了吗？ → CPU 块期间的配置文件显示正在运行的内容

📊 Application Insights 查询

用于诊断Node.js问题的实用KQL查询：

慢速请求

requests
| where timestamp > ago(1h)
| where duration > 5000
| summarize count() by bin(timestamp, 5m), name
| render timechart

未完成的请求

requests
| where timestamp > ago(1h)
| where success == false
| summarize count() by resultCode, name
| order by count_ desc

例外分解

exceptions
| where timestamp > ago(24h)
| summarize count() by type, outerMessage
| order by count_ desc

依赖延迟

dependencies
| where timestamp > ago(1h)
| summarize avg(duration), percentile(duration, 95) by name
| order by avg_duration desc

自定义事件环路度量（如果带了仪器）

customMetrics
| where name == "EventLoopLag"
| summarize avg(value), max(value) by bin(timestamp, 1m)
| render timechart
