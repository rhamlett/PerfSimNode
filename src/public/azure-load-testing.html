📈 Azure Load Testing Guide

Load test endpoint designed for Azure Load Testing integration

🔬 Overview

The Performance Problem Simulator includes a dedicated load testing endpoint designed for use with Azure Load Testing. This endpoint simulates realistic application behavior that degrades gracefully under load.

The /api/loadtest endpoint is designed to:

  • Perform sustained CPU work (interleaved spin-loop cycles at configurable intensity)
  • Hold memory buffers for entire request duration (split between V8 heap and native Buffer for realistic pressure)
  • Degrade gracefully as concurrent requests increase (soft limit pattern)
  • Throw random exceptions when concurrent requests exceed a configurable threshold
  • Eventually trigger Azure's 230-second timeout under extreme load

⚙️ Setting Up Azure Load Testing

  1. In the Azure Portal, create an Azure Load Testing resource
  2. Create a new test and configure the target URL using one of these options:

GET with Query Parameters

Use the target URL directly — no JMeter script required:

# With defaults
https://your-app.azurewebsites.net/api/loadtest

# Custom parameters
https://your-app.azurewebsites.net/api/loadtest?workIterations=2000&bufferSizeKb=50000

# Event loop only (no CPU work)
https://your-app.azurewebsites.net/api/loadtest?workIterations=0&bufferSizeKb=100
  1. Configure load pattern (ramp up, steady state, ramp down)
  2. Set up Azure Monitor integration to correlate metrics

📋 Query Parameters Reference

Each parameter controls a specific resource independently:

Parameter Default Target Resource Description
workIterations 700 CPU Spin-loop iterations per CPU work cycle. ms per cycle = workIterations/10. Default 700 → 70ms spin per cycle.
bufferSizeKb 10000 Memory Memory held for entire request duration in KB (10 MB default). Split 50/50 between V8 heap and native Buffer.
baselineDelayMs 1000 Event Loop Minimum request duration (ms). CPU work and sleep interleaved throughout.
softLimit 20 Concurrency Concurrent requests before degradation begins. Lower = faster degradation.
degradationFactor 1000 Concurrency Additional delay (ms) per request over softLimit.
errorAboveConcurrent 20 Chaos Concurrent request threshold for error injection. When concurrent requests exceed this value, errors may be thrown. Same as softLimit by default. Set to 0 to disable.
errorPercent 20 Chaos Percentage chance (0-100) of throwing a random exception when above errorAboveConcurrent threshold.

📊 Understanding the Soft Limit

The soft limit creates a degradation curve where response times increase as concurrent requests exceed the threshold. The degradation formula is: 

additionalDelay = max(0, concurrent - softLimit) × degradationFactor

Note: Actual response times will vary based on your Azure App Service tier, current load, and other factors. Use Azure Load Testing to establish baseline performance for your specific environment. The default parameters are tuned for a Premium0V3 worker. Any other SKU or worker size will need individual tuning.

Example Degradation Pattern

With default settings (softLimit=20, degradationFactor=1000):

Concurrent Requests Additional Delay Expected Response Time
10 0ms ~1000ms (baseline only)
20 0ms ~1000ms (at soft limit)
30 10,000ms ~11s
50 30,000ms ~31s

💥 Error Injection

The load test endpoint can inject random exceptions to simulate realistic application failures. This is controlled by two parameters:

  • errorAboveConcurrent (default: 20) - Concurrent request threshold before errors may start (same as softLimit)
  • errorPercent (default: 20) - Probability of error once threshold is reached

💡 Concurrency-Based Error Injection: Error injection is enabled by default and triggers when concurrent requests exceed errorAboveConcurrent (default: 20, same as softLimit). This means errors naturally start occurring when the system begins degrading under load. Set errorAboveConcurrent=0 to disable error injection entirely.

Exception Types

Random exceptions are selected from a pool of realistic JavaScript/Node.js exceptions:

Exception Type Example Message
Error InvalidOperationError: Operation is not valid due to current state
TypeError Cannot read properties of null
ReferenceError Object reference not set to an instance of an object
RangeError Index was outside the bounds of the array
Error TimeoutError: The operation has timed out
Error IOException: Unable to read data from the transport connection
Error OutOfMemoryError: Insufficient memory to continue execution

Chaos Testing Examples

# Errors when concurrent requests exceed 10 (50% chance each)
/api/loadtest?errorAboveConcurrent=10&errorPercent=50

# Errors when concurrent requests exceed 25 (rare, 5% chance)
/api/loadtest?errorAboveConcurrent=25&errorPercent=5

# Disable random errors (enabled by default at softLimit)
/api/loadtest?errorAboveConcurrent=0

🔍 Diagnosing Load Test Results

1. Application Insights Integration

  • Go to Application Insights > Performance
  • Filter by operation name GET /api/loadtest
  • Observe response time percentiles (p50, p95, p99)
  • Check failure rates when error injection is enabled

2. Azure Monitor Metrics

  • Requests - Total request count and rate
  • Response Time - Average and percentile latencies
  • Http Server Errors - 5xx errors from random exceptions
  • CPU Percentage - Should increase with load
  • Memory Working Set - Should increase with bufferSizeKb

3. Load Testing Statistics Endpoint

Query the stats endpoint during the test:

GET /api/loadtest/stats

{
  "currentConcurrentRequests": 127,
  "totalRequestsProcessed": 45230,
  "totalExceptionsThrown": 23,
  "averageResponseTimeMs": 847.5
}

📤 Response Format

Each successful request returns:

{
  "elapsedMs": 1247,
  "queueTimeMs": 15,
  "workTimeMs": 1232,
  "concurrentRequestsAtStart": 42,
  "degradationDelayAppliedMs": 8500,
  "workIterationsCompleted": 700,
  "memoryAllocatedBytes": 10240000,
  "workCompleted": true,
  "message": "Load test completed in 1247ms"
}

Response Fields

Field Description
elapsedMs Total request duration in milliseconds (including queue time)
queueTimeMs Time spent waiting in the event loop queue before work began
workTimeMs Actual time spent doing work (elapsedMs - queueTimeMs)
concurrentRequestsAtStart Number of concurrent requests when this request started
degradationDelayAppliedMs Additional delay applied due to exceeding soft limit
workIterationsCompleted Number of CPU work cycles completed
memoryAllocatedBytes Memory buffer size allocated for this request
workCompleted Whether all work completed successfully
message Human-readable completion message

✅ Best Practices

DO:

  • ✅ Start with default parameters to establish a baseline
  • ✅ Monitor Azure metrics during load tests
  • ✅ Use the /api/loadtest/stats endpoint to track concurrency
  • ✅ Gradually increase load to find breaking points
  • ✅ Correlate load test results with Application Insights
  • ✅ Test with realistic load patterns (ramp up, steady state, ramp down)

DON'T:

  • ❌ Start with extreme parameter values
  • ❌ Ignore memory metrics when using large bufferSizeKb
  • ❌ Disable error injection (errorAboveConcurrent=0) if you want to see realistic failures
  • ❌ Assume results from one SKU apply to another
