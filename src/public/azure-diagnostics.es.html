☁️ Azure Guía de diagnóstico

Cómo diagnosticar Node.js problemas de rendimiento usando Azure herramientas y entender qué significan esas métricas

📈 Comprendiendo las métricas del panel de control

El panel de control de PerfSimNode muestra métricas en tiempo real que ayudan a identificar problemas de rendimiento. Esto es lo que significa cada métrica:

⚡ CPU Uso (%)

Lo que mide: Porcentaje del tiempo CPU consumido por el proceso de Node.js.

Por qué es importante: Alto CPU indica operaciones limitadas al cálculo. En Node.js, esto suele significar que se está ejecutando código síncrono (cripto, JSON análisis de objetos grandes, cálculos complejos).

Rango normal: Del 0 al 30% en reposo, los picos durante las peticiones son normales.

Señales de advertencia: Un >70% sostenido indica un problema.

📊 Conjunto de Trabajo de Memoria (MB)

Lo que mide: Memoria total que el proceso de Node.js ha asignado y está usando activamente.

Por qué es importante: La memoria que crece sin liberarse indica fugas. Node.js usa el recogedor de basura del V8, pero las referencias que se mantuvieron demasiado tiempo impiden la limpieza.

Rango normal: Depende de la app, pero debería estabilizarse después del calentamiento.

Señales de advertencia: Crecimiento continuo a lo largo del tiempo (el patrón de dientes de sierra está bien).

🧵 Lag del bucle de eventos (ms)

Lo que mide: Tiempo entre que se programa una callback y que se ejecuta realmente.

Por qué es importante: ¡Esta es la métrica clave para sync-over-async problemas! Cuando el código síncrono bloquea el bucle de eventos, TODAS las devoluciones de llamada pendientes deben esperar, incluidas las solicitudes de HTTP entrantes.

Rango normal: <10ms

Señales de advertencia: >100 ms causa retrasos notables; >1000 ms significa bloqueo severo.

⏱️ Latencia de petición (ms)

Lo que mide: Tiempo desde la solicitud recibida hasta la respuesta enviada, incluyendo el tiempo de espera en la cola del bucle de eventos.

Por qué es importante: Medida directa de la experiencia del usuario. Alta latencia con bajo CPU suele indicar bloqueos o problemas de dependencia externa.

Rango normal: <150 ms para peticiones simples

Señales de advertencia: >1 degradados; >30 críticos.

💡 Información clave: Lag en el bucle de eventosNode.js utiliza un Bucle de eventos de hilo único. Cuando ves un alto retardo en el bucle de eventos, significa que el hilo principal está bloqueado y TODAS las devoluciones de llamada pendientes deben esperar; por eso una sola operación de bloqueo puede afectar a TODAS las solicitudes simultáneamente.

🔍 Escenarios diagnósticos

Así es como diagnosticar cada tipo de problema de rendimiento que crea este simulador:

🔥 Alto uso de CPU

Síntoma:CPU métrica fijada en alto, las solicitudes pueden seguir completándose normalmente (el bucle de eventos no está bloqueado).

¿Qué está pasando?

  • La simulación genera procesos hijos mediante fork()
  • Cada hijo ejecuta CPUcriptografía intensiva (PBKDF2)
  • El trabajo está aislado del bucle de eventos principales: el servidor sigue respondiendo
  • Múltiples node Procesos visibles en top

Perspectiva clave

CPU estrés ≠ bloqueo del bucle de eventos! Un CPU alto en procesos hijos no bloquea el servidor. Observa el lag del bucle de eventos: se mantiene bajo incluso con CPUaltas.

Azure Herramientas de diagnóstico

  • App Service Diagnósticos → CPU Profundizar
  • Application Insights → Rendimiento
  • Kudu SSH → top, ps aux | grep node
  • Perfilado V8 → --prof para identificar funciones calientes

Qué buscar

  • ¿Qué proceso(s) consumen(es) CPU
  • Relación de proceso padre/hijo
  • El lag del bucle de eventos debería mantenerse BAJO
  • Salida del perfil V8 que muestra funciones de alta intensidad
📊 Presión de memoria

Síntoma: La memoria crece, se mantiene alta y puede acabar colapsando con OOM.

¿Qué está pasando?

  • La simulación asigna búferes y contiene referencias
  • El recogedor de basura V8 no puede liberar memoria retenida
  • Simula fugas de memoria por un error de caché

Azure Herramientas de diagnóstico

  • App Service Diagnósticos → Análisis de memoria
  • Application Insights → Métricas
  • Kudu → Explorador de procesos
  • Perfilado V8 → Instantáneas de heap mediante Chrome DevTools

Qué buscar

  • Instantáneas de heap para encontrar objetos retenidos (usar la vista Retainer)
  • Compara instantáneas a lo largo del tiempo para identificar asignaciones crecientes
  • Patrón de crecimiento de memoria (fuga vs. uso normal)
  • Correlación con operaciones específicas
🧵 Bloqueo de bucle de eventos

Síntoma: Picos de retardo en bucle de eventos, TODAS las colas de peticiones, la tabla de latencia muestra latencia alta sostenida en tiempo real.

¿Qué está pasando?

  • La criptografía síncrona se ejecuta DENTRO del hilo principal en bloques repetidos
  • Entre bloques, un breve rendimiento permite que los datos de la sonda en cola se vacien
  • El bucle de eventos se bloquea ~97% de las veces — efectivamente no responde
  • La tabla de latencias se actualiza en tiempo real mostrando el efecto de bloqueo
  • Diferencia clave con CPU estrés: ¡El servidor no responde!

Cómo identificar

¡Comprueba el lag del bucle de eventos! Si es igual a la duración del fragmento, el bucle de eventos queda bloqueado. CPU estrés muestra un alto CPU pero un retardo de bucle de evento MUY Bajo.

Azure Herramientas de diagnóstico

  • Application Insights → Rendimiento
  • App Service Diagnósticos → Aplicación web lenta
  • Azure Monitor → Métricas para picos de latencia de peticiones
  • Perfilado V8 → CPU perfil durante el bloqueo muestra código de bloqueo

Qué buscar

  • Retardo del bucle de eventos = duración del fragmento (~200ms por defecto)
  • TODAS las solicitudes afectadas simultáneamente
  • El gráfico de latencia muestra latencia sostenida elevada durante la simulación
  • CPU puede ser alta O baja (depende del trabajo de bloqueo)
  • El perfil V8 identifica la función síncrona (por ejemplo, pbkdf2Sync)
🐢 Solicitudes lentas

Síntoma: Un endpoint específico tarda mucho, pero otros endpoints son NORMALES.

¿Qué está pasando?

  • Usos del gestor de solicitudes setTimeout() por retraso
  • No bloqueo: el bucle de eventos continúa procesándose
  • Simula llamadas de base de datos y API externas lentas
  • Diferencia clave respecto al bloqueo de bucles de eventos: ¡Solo afecta el final lento!

Cómo identificar

Comprueba si OTRAS solicitudes se ven afectadas. Solicitudes lentas: solo el endpoint lento es lento. Bloqueo de bucle de eventos: TODO es lento.

Azure Herramientas de diagnóstico

  • Application Insights → Transacción de extremo a extremo
  • Application Insights → Seguimiento de dependencias
  • Análisis de registros → duración de la solicitud por endpoint

Qué buscar

  • ¿Qué punto final específico es lento?
  • Desglose temporal de dependencias
  • El lag del bucle de eventos se mantiene BAJO
💥 Fallo de la aplicación

Síntoma: La app deja de responder, se reinicia o devuelve errores 502/503.

¿Qué está pasando?

  • Proceso de fallos de excepciones no gestionadas
  • OOM asesino termina el proceso
  • App Service puede reiniciar automáticamente

Azure Herramientas de diagnóstico

  • App Service Diagnósticos → Se ha reiniciado la aplicación web
  • Application Insights → Fracasos
  • Kudu → Archivos de registro
  • Perfilado V8 → Instantánea del heap antes del fallo para identificar la causa

Qué buscar

  • Análisis de volcados de emergencia
  • Traza de pila de excepciones en los registros
  • Uso de memoria antes del fallo (comparación de snapshots de heap)

🔗 Correlación ID de simulación

A cada instancia de simulación se le asigna un ID de Simulación único (GUID) que permite una correlación extremo a extremo entre la interfaz del panel de control y los diagnósticos de Azure .

Encontrar identificadores de simulación

En el panel de control: Pasa el cursor sobre cualquier mensaje relacionado con la simulación en el Registro de Eventos (marcado con un subrayado punteado) para ver el ID completo de la Simulación en una descripción emergente.

En Azure: El mismo ID de simulación se emite en los registros del lado del servidor y puede encontrarse en:

  • Application Insights - Trazas de búsqueda y eventos personalizados por ID de simulación
  • Análisis de registros - Registros de consulta usando el ID como filtro
  • Contenedor/App Service troncos - Filtrar stdout/stderr por el GUID

Flujo de trabajo de resolución de problemas

  1. Inicia una simulación desde el panel de control
  2. Pasa el cursor sobre el mensaje de inicio en el Registro de Eventos para copiar el ID de Simulación
  3. Portal abierto Azure → Application Insights → Registros (o Análisis de Registros)
  4. Consulta los registros usando el ID de simulación para ver toda la actividad relacionada del lado del servidor
  5. Correlacionar las marcas de tiempo entre eventos del panel de control y métricas del servidor
💡 Propina

El ID de Simulación ayuda a aislar la telemetría al ejecutar múltiples simulaciones simultáneas o cuando el panel de control es utilizado por varios usuarios.

🛠️ Azure Herramientas de diagnóstico

Herramientas clave Azure para diagnosticar problemas de rendimiento Node.js:

App Service Diagnóstico

Informes diagnósticos integrados en Azure Portal. Acceso a través de "Diagnosticar y resolver problemas".

Lo mejor para: Resumen rápido de la salud, análisis de CPU/memoria, investigación de fallos.

Application Insights

Solución APM completa con rastreo de solicitudes, seguimiento de dependencias y métricas personalizadas.

Lo mejor para: Rastreo de extremo a extremo, tendencias de rendimiento, análisis de fallos.

Kudu (SCM)

Consola para desarrolladores en yourapp.scm.azurewebsites.net

Lo mejor para: Depuración en tiempo real, acceso a registros, gestión de procesos.

Análisis de registros

Análisis de logs basado en consultas con KQL.

Lo mejor para: Análisis histórico, consultas personalizadas, alertas.

Azure Monitorizar métricas

Métricas de plataforma para CPU, memoria, peticiones, etc.

Lo mejor para: Monitorización en tiempo real, paneles, alertas.

Análisis de volcado de choque

Recogida automática o manual de volcado para la investigación de accidentes.

Lo mejor para: Análisis de caídas post mortem, investigación de fugas de memoria.

🔄 Flujo de trabajo de diagnóstico

Sigue este flujo de trabajo al investigar un problema de rendimiento Node.js:

Informe de clientes → Comprobar métricas → Identificar patrón → Inmersión profunda → Causa raíz

Paso 1: Evaluación inicial

  1. Consulta App Service Resumen para ver el estado de salud
  2. Mira métricas de CPU, Memoria, Peticiones en Azure Monitor
  3. Fíjate en el periodo de duración del número

Paso 2: Identificar el patrón

Si ves... Probable problema Siguiente paso
Altos CPU, tiempos de respuesta normales Procesamiento CPUlimitado a - (esperado o con error) CPU perfilado vía Kudu
Respuestas altas CPUy lentas CPU la inanición afecta a las solicitudes Identificar la ruta de código caliente
Respuestas CPUnormales, lentas TODAS Bloqueo de bucle de eventos (sync-over-async) Revisa métricas de bucles de eventos, revisión de código
Respuestas CPUnormales, lentas ALGUNAS Latencia de dependencia externa Application Insights Vista de dependencias
La memoria crece con el tiempo Fuga de memoria Análisis de instantáneas de heap
Reinicios repentinos, errores 502 Bloqueo (excepción o OOM) Volcado de fallo, registros de errores

Paso 3: Análisis profundo de herramientas por número

🔥 ¡Alto CPU
  1. Kudu Process Explorer → proceso Identify
  2. App Service Diagnósticos → CPU Detalle
  3. Perfil con node --prof o --inspect
🧵 Bloqueo de bucle de eventos
  1. Consulta Application Insights solicitudes de lentitud correlacionadas
  2. Busca operaciones de sincronización en el código (fs.readFileSync, etc.)
  3. Añadir monitorización con perf_hooks.monitorEventLoopDelay()
📊 Fuga de memoria
  1. Toma una instantánea de heap mediante Kudu o depuración remota
  2. Comparar instantáneas a lo largo del tiempo
  3. Busca el aumento del número de objetos

🔬 Perfilado V8

V8 es el motor de JavaScript que impulsa Node.js. Incluye herramientas de perfilado integradas que ayudan a identificar cuellos de botella de rendimiento, fugas de memoria y rutas de código CPUintensivas.

¿Qué es el perfilado V8?

El perfilado V8 recoge información detallada sobre el comportamiento en tiempo de ejecución de tu aplicación:

  • CPU Perfilado — Registros de qué funciones consumen más CPU tiempo, ayudando a identificar rutas de código activo y oportunidades de optimización
  • Instantáneas de heap — Captura el estado de la memoria en un momento determinado, mostrando todos los objetos asignados y sus referencias
  • Cronología del montón — Rastrea la asignación de memoria a lo largo del tiempo para identificar fugas y exceso de recogida de basura

Método 1: Perfilado por línea de comandos (--prof)

El --prof Flag genera un perfil detallado de CPU que muestra el tiempo dedicado a cada función.

Paso 1: Ejecutar con el perfilado activado

# Start your app with profiling
node --prof app.js

# Reproduce the performance issue, then stop the app (Ctrl+C)
# This creates an isolate-*.log file in the current directory

Paso 2: Procesar el perfil

# Convert the V8 log to human-readable format
node --prof-process isolate-0x*.log > profile.txt

Paso 3: Lee los resultados

El perfil procesado muestra un desglose así:

[Summary]:
   ticks  total  nonlib   name
   5234   75.2%   78.1%  JavaScript
   1023   14.7%   15.3%  C++
    445    6.4%    6.6%  GC
    258    3.7%          Shared libraries

[JavaScript]:
   ticks  total  nonlib   name
   3102   44.6%   46.3%  pbkdf2Sync
    892   12.8%   13.3%  JSON.parse
    ...

Busca funciones con un alto número de ticks: estos son tus puntos de CPU calientes.

Método 2: Chrome DevTools (--inspección)

Para perfiles interactivos con interfaz visual, utiliza el inspector de DevTools de Chrome.

Paso 1: Empieza con Inspector

# Local development
node --inspect app.js

# Allow remote connections (e.g., from Azure)
node --inspect=0.0.0.0:9229 app.js

Paso 2: Conecta Chrome DevTools

  1. Abre Chrome y navega a chrome://inspect
  2. Haz clic en "Configurar..." y añadir localhost:9229 (o tu anfitrión remoto)
  3. Tu proceso Node.js aparece en "Objetivo remoto"—haz clic en "inspeccionar"

Paso 3: Captura un perfil CPU

  1. Ve a la Perfilador pestaña (o pestaña Rendimiento)
  2. Clic Comienzo
  3. Reproduce el problema de rendimiento
  4. Clic ¡Para
  5. Analiza el gráfico de llamas: las barras anchas indican funciones lentas

Paso 4: Haz una instantánea del montón

  1. Ve a la Memoria pestaña
  2. Selecciona "Instantánea del heap" y haz clic Haz una instantánea
  3. Explora la vista "Resumen" para ver el número y tamaño de objetos
  4. Compara múltiples instantáneas para identificar fugas de memoria

Método 3: Instantáneas programáticas del montón

Genera instantáneas de heap desde dentro de tu código o mediante señales:

// In your application code
const v8 = require('v8');
const fs = require('fs');

function takeHeapSnapshot() {
  const filename = `heap-${Date.now()}.heapsnapshot`;
  const stream = fs.createWriteStream(filename);
  v8.writeHeapSnapshot(filename);
  console.log(`Heap snapshot written to ${filename}`);
}

// Trigger on SIGUSR2 signal
process.on('SIGUSR2', takeHeapSnapshot);

Luego se activa con: kill -USR2 <pid>

Lectura de instantáneas de heaps

Abierto .heapsnapshot archivos en Chrome DevTools:

  1. Abre Chrome DevTools (F12)
  2. Ir a Memoria pestaña
  3. Clic Carga y selecciona tu .heapsnapshot archivo

Puntos clave para analizar:

  • Resumen — Objetos agrupados por nombre del constructor. Busca conteos inesperadamente altos.
  • Comparación — Diferencia dos instantáneas para ver qué se asignó entre ellas.
  • Contención — Muestra la jerarquía de objetos (qué contiene referencias a qué).
  • Sirvientes — Para cualquier objeto, muestra POR QUÉ sigue en memoria (la cadena de referencia a GC raíces).

Azure- Consideraciones específicas

  • Kudu SSH — Ejecutar comandos de perfilado directamente mediante yourapp.scm.azurewebsites.net
  • Reenvío de puertos — Para remotos --inspect, tendrás que configurar el reenvío de puertos a través de Azure CLI o VS Code
  • Descarga de archivos — Utilizar el navegador de archivos de Kudu o az webapp log download para recuperar archivos de perfil
  • Impacto en la producción — El perfilado añade costes generales; Uso en escenografía o captura brevemente en producción
💡 Referencia rápida
  • CPU lento? → Uso --prof o pestaña Perfilador para encontrar funciones calientes
  • ¿La memoria va creciendo? → Haz instantáneas de heap y compara a lo largo del tiempo
  • ¿Bucle de eventos bloqueado? → CPU perfil durante el bloque muestra lo que está funcionando

📊 Application Insights Consultas

Consultas útiles de KQL para diagnosticar Node.js problemas:

Solicitudes lentas

requests
| where timestamp > ago(1h)
| where duration > 5000
| summarize count() by bin(timestamp, 5m), name
| render timechart

Solicitudes fallidas

requests
| where timestamp > ago(1h)
| where success == false
| summarize count() by resultCode, name
| order by count_ desc

Desglose de las excepciones

exceptions
| where timestamp > ago(24h)
| summarize count() by type, outerMessage
| order by count_ desc

Latencia de dependencia

dependencies
| where timestamp > ago(1h)
| summarize avg(duration), percentile(duration, 95) by name
| order by avg_duration desc

Métrica de bucle de eventos personalizada (si está instrumentada)

customMetrics
| where name == "EventLoopLag"
| summarize avg(value), max(value) by bin(timestamp, 1m)
| render timechart
