☁️ Azure Guia de Diagnóstico

Como diagnosticar Node.js problemas de desempenho usando ferramentas Azure e entender o que essas métricas significam

📈 Entendendo as Métricas do Painel

O painel PerfSimNode exibe métricas em tempo real que ajudam a identificar problemas de desempenho. Veja o que cada métrica significa:

⚡ CPU Uso (%)

O que ele mede: Percentual do tempo CPU consumido pelo processo Node.js.

Por que isso importa: Alta CPU indica operações limitadas ao cálculo. Em Node.js, isso normalmente significa que código síncrono está rodando (cripto, JSON análise de objetos grandes, cálculos complexos).

Faixa normal: 0-30% em idle, picos durante as requisições são normais.

Sinais de alerta: Sustentado >70% indica um problema.

📊 Conjunto de Trabalho de Memória (MB)

O que ele mede: Memória total que o processo Node.js alocou e está usando ativamente.

Por que isso importa: Memória que cresce sem liberar indica vazamentos. Node.js usa o coletor de lixo do V8, mas referências mantidas por muito tempo impedem a limpeza.

Faixa normal: Depende do aplicativo, mas deve estabilizar após o aquecimento.

Sinais de alerta: Crescimento contínuo ao longo do tempo (padrão dente de serra é ok).

🧵 Atraso do Loop de Eventos (ms)

O que ele mede: Tempo entre o agendamento de um callback e a execução efetiva.

Por que isso importa: Essa é a métrica chave para sync-over-async problemas! Quando o código síncrono bloqueia o loop de eventos, TODAS as chamadas pendentes devem aguardar—incluindo as solicitações de HTTP recebidas.

Faixa normal: <10ms

Sinais de alerta: >100ms causa atrasos perceptíveis; >1000ms significa bloqueio severo.

⏱️ Latência de Solicitação (ms)

O que ele mede: Tempo desde o pedido recebido até a resposta enviada, incluindo o tempo de espera na fila do loop de eventos.

Por que isso importa: Medida direta da experiência do usuário. Alta latência com baixa CPU frequentemente indica bloqueio ou problemas de dependência externa.

Faixa normal: <150ms para solicitações simples

Sinais de alerta: >1 degradados; >30 anos crítico.

💡 Insight Chave: Atraso no Loop de EventosNode.js usa um Loop de eventos de thread única. Quando você vê um alto Event Loop Lag, significa que a thread principal está bloqueada e TODOS os callbacks pendentes devem esperar — é por isso que uma única operação de bloqueio pode afetar TODAS as requisições simultaneamente.

🔍 Cenários Diagnósticos

Veja como diagnosticar cada tipo de problema de desempenho que este simulador cria:

🔥 Alto Uso CPU

Sintoma:CPU métrica fixada em alto, as requisições ainda podem ser concluídas normalmente (o loop de eventos não está bloqueado).

O que está acontecendo

  • A simulação gera processos filhos via fork()
  • Cada filho executa CPUcriptografia intensiva (PBKDF2)
  • O trabalho é isolado do ciclo de evento principal - o servidor permanece responsivo
  • Múltiplos node Processos visíveis em top

Principais Insights

CPU estresse ≠ bloqueio de loop de eventos! Alta CPU em processos filhos não bloqueia o servidor. Cuidado com o Event Loop Lag – ele permanece baixo mesmo com CPUalto.

Azure Ferramentas de Diagnóstico

  • App Service Diagnósticos → CPU Aprofundar
  • Application Insights → Desempenho
  • Kudu SSH → top, ps aux | grep node
  • Perfilamento V8 → --prof para identificar funções quentes

O que procurar

  • Qual(es) processo(s) consome CPU
  • Relação de processo pai/filho
  • O atraso do loop de eventos deve permanecer BAIXO
  • Saída do perfil V8 mostrando funções de alto tick
📊 Pressão de Memória

Sintoma: A memória cresce, permanece alta e pode eventualmente desmoronar com OOM.

O que está acontecendo

  • A simulação aloca buffers e mantém referências
  • Coletor de lixo V8 não consegue liberar memória armazenada
  • Simula vazamentos de memória devido a cache que deram errado

Azure Ferramentas de Diagnóstico

  • App Service Diagnósticos → Análise de Memória
  • Application Insights → Métricas
  • Kudu → Explorador de Processos
  • Perfilamento V8 → Snapshots de heap via Chrome DevTools

O que procurar

  • Snapshots de heap para encontrar objetos retidos (use a visualização Retainer)
  • Compare instantâneos ao longo do tempo para identificar alocações crescentes
  • Padrão de crescimento de memória (vazamento vs. uso normal)
  • Correlação com operações específicas
🧵 Bloqueio de Loop de Eventos

Sintoma: Picos de lag no ciclo de eventos, fila de TODAS as requisições, gráfico de latência mostra latência alta sustentada em tempo real.

O que está acontecendo

  • Criptomoedas síncronas rodam DENTRO do thread principal em blocos repetidos
  • Entre blocos, um rendimento breve permite que dados de sonda em fila esvazem
  • O loop de eventos é bloqueado ~97% das vezes — efetivamente não responsivo
  • O gráfico de latência é atualizado em tempo real mostrando o efeito de bloqueio
  • Diferença chave em relação ao estresse CPU : O servidor não responde!

Como Identificar

Verifique o Lag do Loop de Eventos! Se for igual à duração do chunk, o loop de eventos é bloqueado. CPU estresse mostra alta CPU , mas BAIXO atraso no loop de eventos.

Azure Ferramentas de Diagnóstico

  • Application Insights → Desempenho
  • App Service Diagnósticos → Aplicativo Web Lento
  • Azure Monitor → Métricas para picos de latência de requisição
  • Perfilamento V8 → CPU perfil durante o bloco mostra código de bloqueio

O que procurar

  • Atraso do loop de evento = duração do chunk (~200ms padrão)
  • TODAS as solicitações afetadas simultaneamente
  • O gráfico de latência mostra latência elevada sustentada durante a simulação
  • CPU pode ser alto OU baixo (depende do trabalho de bloqueio)
  • O perfil V8 identifica a função síncrona (por exemplo, pbkdf2Sync)
🐢 Pedidos Lentos

Sintoma: Endpoints específicos demoram muito, mas outros endpoints são NORMAIS.

O que está acontecendo

  • Usos do handler de requisições setTimeout() para atraso
  • Não bloqueio: o ciclo de eventos continua processando
  • Simula chamadas lentas de banco de dados e API externas
  • Diferença chave em relação ao Bloqueio de Loop de Eventos: Só o endpoint lento é afetado!

Como Identificar

Verifique se OUTROS pedidos estão afetados. Requisições lentas: só o endpoint lento é lento. Bloqueio de loop de eventos: TUDO é lento.

Azure Ferramentas de Diagnóstico

  • Application Insights → Transação de ponta a ponta
  • Application Insights → Rastreamento de dependências
  • Análise de Logs → Duração da solicitação por endpoint

O que procurar

  • Qual endpoint específico é lento
  • Quebra de temporização de dependências
  • O atraso do loop de eventos permanece BAIXO
💥 Crash de Aplicação

Sintoma: O aplicativo fica sem resposta, reinicia ou retorna erros 502/503.

O que está acontecendo

  • Processo de falhas de exceção não tratadas
  • OOM assassino encerra o processo
  • App Service pode reiniciar automaticamente

Azure Ferramentas de Diagnóstico

  • App Service Diagnósticos → App Web Reiniciado
  • Application Insights → Falhas
  • Kudu → LogFiles
  • Perfilamento V8 → snapshot de heap antes do travamento para identificar a causa

O que procurar

  • Análise de despejo de emergência
  • Rastreamento de pilha de exceção em logs
  • Uso de memória antes do crash (comparação de snapshots de heap)

🔗 Correlação do ID da Simulação

Cada instância de simulação recebe um ID de Simulação único (GUID) que permite correlação de ponta a ponta entre a interface do painel e os diagnósticos Azure .

Encontrando IDs de Simulação

No Painel: Passe o mouse sobre qualquer mensagem relacionada à simulação no Registro de Eventos (marcada com um sublinhado pontilhado) para ver o ID completo da Simulação em uma dica de ferramenta.

Em Azure: O mesmo ID de simulação é emitido nos logs do lado do servidor e pode ser encontrado em:

  • Application Insights - Rastreamentos de busca e eventos personalizados por ID de Simulação
  • Análise de Logs - Logs de consulta usando o ID como filtro
  • Troncos de Contêineres/App Service - Filtre stdout/stderr pelo GUID

Fluxo de Trabalho de Solução de Problemas

  1. Inicie uma simulação a partir do painel
  2. Passe o mouse sobre a mensagem de início no Log de Eventos para copiar o ID da Simulação
  3. Abrir Azure Portal → Application Insights → Logs (ou Análise de Logs)
  4. Consulte logs usando o ID de simulação para ver toda a atividade relacionada do lado do servidor
  5. Correlacione carimbos de tempo entre eventos do painel e métricas de servidor
💡 Dica

O ID de Simulação ajuda a isolar a telemetria ao executar múltiplas simulações simultâneas ou quando o painel é usado por múltiplos usuários.

🛠️ Azure Ferramentas de Diagnóstico

Principais ferramentas Azure para diagnosticar problemas de desempenho Node.js:

App Service Diagnóstico

Relatórios diagnósticos integrados no Azure Portal. Acesso via "Diagnosticar e resolver problemas".

Melhor para: Visão geral rápida da saúde, análise CPU/memória, investigação de falhas.

Application Insights

Solução completa de APM com rastreamento de requisições, rastreamento de dependências e métricas personalizadas.

Melhor para: Rastreamento de ponta a ponta, tendências de desempenho, análise de falhas.

Kudu (SCM)

Console de desenvolvedor em yourapp.scm.azurewebsites.net

Melhor para: Depuração em tempo real, acesso a logs, gerenciamento de processos.

Análise de Logs

Análise de logares baseada em consulta com KQL.

Melhor para: Análise histórica, consultas personalizadas, alertas.

Azure Métricas de Monitoramento

Métricas de plataforma para CPU, memória, requisições, etc.

Melhor para: Monitoramento em tempo real, painéis, alertas.

Análise de Despejo de Acidentes

Coleta automática ou manual de despejo para investigação de acidentes.

Melhor para: Análise de acidentes post-mortem, investigação de vazamento de memória.

🔄 Fluxo de Trabalho de Diagnóstico

Siga este fluxo de trabalho ao investigar um problema de desempenho Node.js:

Relatório de Clientes → Verifique as métricas → Identificar Padrão → Mergulho profundo → Causa Raiz

Passo 1: Avaliação Inicial

  1. Consulte App Service Visão Geral para o estado de saúde
  2. Veja métricas CPU, Memória, Solicitação no Azure Monitor
  3. Note o período de tempo da edição

Passo 2: Identifique o Padrão

Se você vê... Provável Problema Próximo passo
Tempo de resposta alto CPUe normal Processamento CPUligado a - (esperado ou bug) CPU perfiling via Kudu
Respostas CPUaltas e lentas CPU a fome afetando os pedidos Identificar caminho de código quente
CPUnormal, TODAS as respostas lentas Bloqueio de loop de eventos (sync-over-async) Verifique métricas de loop de eventos, revisão de código
Respostas CPUnormais, lentas ALGUMAS Latência de dependência externa Application Insights visão de dependência
Memória crescendo com o tempo Vazamento de memória Análise de snapshots de heap
Reiniciações repentinas, erros 502 Travamento (exceção ou OOM) Despejo de falha, logs de erro

Passo 3: Aprofunde as Ferramentas por Edição

🔥 Alto CPU
  1. Kudu Process Explorer → Identificar processo
  2. App Service Diagnósticos → CPU Aprofundamento
  3. Perfil com node --prof ou --inspect
🧵 Bloqueio de Loop de Eventos
  1. Verifique Application Insights para solicitações de lentidão correlacionadas
  2. Procure operações de sincronização no código (fs.readFileSync, etc.)
  3. Adicionar monitoramento com perf_hooks.monitorEventLoopDelay()
📊 Vazamento de Memória
  1. Faça snapshot de heap via Kudu ou depuração remota
  2. Compare instantâneos ao longo do tempo
  3. Procure o aumento da contagem de objetos

🖥️ Comandos do Console Kudu

Comandos úteis quando conectado ao console Kudu (SCM):

Informações de Processo

# List Node.js processes
ps aux | grep node

# Show memory usage
free -m

# CPU and memory in real-time
top -p $(pgrep -f node)

Perfilamento

# Generate CPU profile (run this, reproduce issue, stop with Ctrl+C)
# Profile saved as isolate-*.log
node --prof /home/site/wwwroot/index.js

# Process the profile
node --prof-process isolate-*.log > profile.txt

Heap Snapshot

# Find Node.js PID
pgrep -f node

# Send signal to generate heap snapshot
kill -USR2 <pid>

# Or connect with Chrome DevTools
# 1. In Kudu, run: node --inspect=0.0.0.0:9229 /home/site/wwwroot/index.js
# 2. Set up port forwarding
# 3. Connect Chrome to chrome://inspect

Análise de Logs

# View recent application logs
tail -100 /home/LogFiles/Application/*.log

# Search for errors
grep -i "error\|exception" /home/LogFiles/Application/*.log

# Watch logs in real-time
tail -f /home/LogFiles/Application/*.log

🔬 Perfilamento V8

V8 é o motor JavaScript que alimenta Node.js. Inclui ferramentas de perfilamento integradas que ajudam a identificar gargalos de desempenho, vazamentos de memória e caminhos de código intensivos CPU.

O que é Perfilamento V8?

O perfilamento V8 captura informações detalhadas sobre o comportamento em tempo de execução da sua aplicação:

  • CPU Perfilamento — Registros de quais funções consomem mais tempo CPU , ajudando a identificar caminhos de código quente e oportunidades de otimização
  • Heap Snapshots — Captura o estado da memória em um momento específico, mostrando todos os objetos alocados e suas referências
  • Linha do tempo do heap — Acompanha a alocação de memória ao longo do tempo para identificar vazamentos e coleta excessiva de lixo

Método 1: Perfil de Linha de Comando (--prof)

O --prof Flag gera um perfil detalhado de CPU que mostra o tempo gasto em cada função.

Passo 1: Execute com Perfilamento Ativado

# Start your app with profiling
node --prof app.js

# Reproduce the performance issue, then stop the app (Ctrl+C)
# This creates an isolate-*.log file in the current directory

Passo 2: Processe o Perfil

# Convert the V8 log to human-readable format
node --prof-process isolate-0x*.log > profile.txt

Passo 3: Leia os Resultados

O perfil processado mostra uma divisão assim:

[Summary]:
   ticks  total  nonlib   name
   5234   75.2%   78.1%  JavaScript
   1023   14.7%   15.3%  C++
    445    6.4%    6.6%  GC
    258    3.7%          Shared libraries

[JavaScript]:
   ticks  total  nonlib   name
   3102   44.6%   46.3%  pbkdf2Sync
    892   12.8%   13.3%  JSON.parse
    ...

Procure funções com alta contagem de ticks — essas são suas CPU hotspots.

Método 2: Chrome DevTools (--inspect)

Para perfilamento interativo com interface visual, use o inspetor DevTools do Chrome.

Passo 1: Comece com o Inspetor

# Local development
node --inspect app.js

# Allow remote connections (e.g., from Azure)
node --inspect=0.0.0.0:9229 app.js

Passo 2: Conecte o Chrome DevTools

  1. Abra o Chrome e navegue até chrome://inspect
  2. Clique em "Configurar..." e adiciona localhost:9229 (ou seu host remoto)
  3. Seu processo Node.js aparece em "Alvo Remoto"—clique em "inspecionar"

Passo 3: Capture um Perfil CPU

  1. Vá para o Perfilador aba (ou aba Performance)
  2. Click Comece
  3. Reproduzir o problema de desempenho
  4. Click Pare
  5. Analise o gráfico de chamas — barras largas indicam funções lentas

Passo 4: Tire uma foto de heap

  1. Vá para o Memória Tab
  2. Selecione "Heap snapshot" e clique Tire uma foto
  3. Explore a visualização "Resumo" para ver a contagem e tamanhos dos objetos
  4. Compare múltiplos snapshots para identificar vazamentos de memória

Método 3: Snapshots Programáticos de Heap

Gere snapshots de heap a partir do seu código ou via sinais:

// In your application code
const v8 = require('v8');
const fs = require('fs');

function takeHeapSnapshot() {
  const filename = `heap-${Date.now()}.heapsnapshot`;
  const stream = fs.createWriteStream(filename);
  v8.writeHeapSnapshot(filename);
  console.log(`Heap snapshot written to ${filename}`);
}

// Trigger on SIGUSR2 signal
process.on('SIGUSR2', takeHeapSnapshot);

Depois, acione com: kill -USR2 <pid>

Leitura de Snapshots de Heap

Aberto .heapsnapshot arquivos no Chrome DevTools:

  1. Abra o Chrome DevTools (F12)
  2. Vá para Memória Tab
  3. Click Carga e selecione seu .heapsnapshot arquivo

Principais pontos de vista para analisar:

  • Resumo — Objetos agrupados por nome do construtor. Procure contagens inesperadamente grandes.
  • Comparação — Diferencia dois instantâneos para ver o que foi alocado entre eles.
  • Contenção — Mostra a hierarquia de objetos (o que está contendo referências a quê).
  • Criados — Para qualquer objeto, mostra POR QUE ele ainda está na memória (a cadeia de referência para GC raízes).

Azure- Considerações Específicas

  • Kudu SSH — Executar comandos de perfilamento diretamente via yourapp.scm.azurewebsites.net
  • Reencaminhamento de Portas — Para remotos --inspect, você precisará configurar o encaminhamento de portas via CLI Azure ou VS Code
  • Download de arquivo — Use o navegador de arquivos de Kudu ou az webapp log download para recuperar arquivos de perfil
  • Impacto na Produção — A perfilagem adiciona custos operacionais; Uso em encenação ou captura brevemente em produção
💡 Referência Rápida
  • CPU devagar? → Uso --prof ou aba Profiler para encontrar funções quentes
  • Memória crescendo? → Tire snapshots de heap e compare ao longo do tempo
  • Loop de evento bloqueado? → CPU perfil durante o bloco mostra o que está rodando

📊 Application Insights Consultas

Consultas úteis no KQL para diagnosticar Node.js problemas:

Pedidos Lentos

requests
| where timestamp > ago(1h)
| where duration > 5000
| summarize count() by bin(timestamp, 5m), name
| render timechart

Pedidos Fracassados

requests
| where timestamp > ago(1h)
| where success == false
| summarize count() by resultCode, name
| order by count_ desc

Distribuição de exceções

exceptions
| where timestamp > ago(24h)
| summarize count() by type, outerMessage
| order by count_ desc

Latência de Dependência

dependencies
| where timestamp > ago(1h)
| summarize avg(duration), percentile(duration, 95) by name
| order by avg_duration desc

Métrica de Loop de Eventos Personalizada (se instrumentada)

customMetrics
| where name == "EventLoopLag"
| summarize avg(value), max(value) by bin(timestamp, 1m)
| render timechart
